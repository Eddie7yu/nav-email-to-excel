# 使用流程：从零到每周自动更新净值表

> 你不需要会编程。所有命令由 AI 助手执行；你只需在本机输入一次邮箱授权码、检查
> 第一次结果。之后由 AI 自动更新。

## 它帮你做什么

AI 从邮箱发现净值发件人和产品 → 与已有工作簿自动匹配，或在没有工作簿时按内置脱敏模板创建 → 生成第一次预览给你检查 →
你批准后写入并开启自动更新。以后每次自动核对、备份和写入，不再重复看预览。

数据冲突、日期缺口或表结构变化时会停止并保留原表，其余能从邮箱和工作簿判断的事情都由 AI 完成。

仓库已经提供常见问题的成熟处理方式，但现实中的邮件和表格格式不可能全部预先列出。遇到新情况时，AI 可以在本机使用副本、临时诊断脚本或本地解析器自主解决；你不需要替它选择技术方案。无论它怎样探索，第一次结果仍由你检查，正式写入仍会验证、备份，失败时不改原表。

## 第 0 步：你只要告诉 AI 两项信息

- 已有净值工作簿的完整路径（`.xlsx` / `.xlsm`）；如果还没有，就告诉 AI 希望新建的 `.xlsx` 完整路径。
- 收净值邮件的邮箱账号。

发件人、产品代码、工作表映射、列号、邮件到达时间和 IMAP 服务器地址都不需要你整理。AI 会先从现有设置和仓库内参考资料确定 IMAP 地址，登录后再从邮箱与工作簿自动发现其余配置。

首次配置过程中，AI 还会给出一个自动运行时间建议，并请你确认更新频率和时间。这是部署所需的唯一业务偏好，例如“工作日每天 18:00”或“每周五 17:30”。

## 运行环境（由 AI 检查）

软硬件：

- [ ] Windows 10/11 电脑（正式写入只支持 Windows；Mac/Linux 只能做到预览）
- [ ] Microsoft Excel 或 WPS 表格（任一）
- [ ] Python 3.11–3.14（python.org 下载，安装时勾选 "Add python.exe to PATH"）
- [ ] Git（git-scm.com，一路默认安装即可）
- [ ] AI 编程助手：Codex、Claude Code 或 Cursor 任一

如果缺少 Python、Git 或可用的 Excel/WPS，AI 会直接指出缺哪一项并给出最短处理办法。你还需要在邮箱网页设置中开启 IMAP 并生成一次 **IMAP 授权码**（不是登录密码）；到第 3 步时在本机隐藏输入即可。

> 邮箱兼容性：支持标准 IMAP over SSL。v1.2.1 起，163、126、yeah.net 和网易
> 企业邮主机会在登录后自动发送所需的 IMAP ID；仅限 OAuth、没有授权码登录方式的
> 邮箱仍暂不支持。

## 第 1 步：把 skill 装给你的 AI

在 PowerShell 里执行（三选一）：

```powershell
# Codex
git clone https://github.com/Eddie7yu/nav-email-to-excel "$env:USERPROFILE\.codex\skills\nav-email-to-excel"
codex

# Claude Code
git clone https://github.com/Eddie7yu/nav-email-to-excel "$env:USERPROFILE\.claude\skills\nav-email-to-excel"
claude
```

Cursor 用户：克隆到任意本地目录，让 Agent 完整阅读其中的 `SKILL.md`（详见仓库内
`references/portable-use.md`）。

```text
请使用 nav-email-to-excel skill，完整阅读 SKILL.md 和 references/configuration.md，
为我的工作簿部署净值邮件自动化：<已有或希望新建的工作簿完整路径>。
如果该文件还不存在，请用内置完全脱敏模板创建；如果已经存在，必须保持原样，不要套模板、改表头或重排。

不要让我手工提供发件人邮箱、产品代码或逐项填写路由。请主动弹出可见授权码窗口，
我只在窗口中隐藏粘贴一次；保存后请自行检查，再自动扫描净值邮件、匹配工作簿并完成配置。第一次生成预览给我检查；
有新增时我批准后写入，零新增时我批准基线报告并直接启用自动更新。后续按邮件到达规律自动备份和更新。只有真正无法从历史证据
判断的业务歧义才问我。
```

## 第 3 步：跟着 AI 走完部署（你在每一步做什么）

| 阶段 | AI 做什么 | 你做什么 |
|---|---|---|
| 1. 创建运行目录 | 默认在工作簿旁创建独立的 `净值自动化\`，运行引导脚本并装好隔离环境；国内网络直连 PyPI 较慢时，只为本次安装启用清华镜像 | 通常无需操作；默认路径过长、不可写或已有同名目录时由 AI 另选新的短路径 |
| 1.5 整理用户入口 | 根目录只保留中文说明、授权/状态/手动更新入口、保护本地资料的 `.gitignore` 和预览、备份、日志目录；程序、配置和环境收进 `app\` | 不需要进入或理解 `app\`，也不要修改 `.gitignore`；日常只使用根目录中文入口 |
| 2. 离线演练 | 用完全虚构的邮件和表格把全流程走一遍 | AI 自己完成；这一步不碰你的真实邮箱和表格 |
| 3. 保存授权码 | 主动弹出独立、可见的授权码输入窗口，并在你输入后自动检查保存状态 | ★ 这是你唯一必须亲自操作的一步：在弹出的窗口右键粘贴授权码（屏幕只显示星号）并回车；不需要自己打开 PowerShell、切换目录或输入命令 |
| 4. 自动配置 | 先只读最近邮件的发件人和主题，再按单主题、短窗口逐产品下载；CLI 显示当前阶段、邮件数和附件数，超出批次或时间预算时保存游标后继续。已有表先只读区分可直接接管与基准待核实产品；没有表则判断日/周频率和可靠基准 | 不需要抄发件人邮箱或判断程序是否“卡死”；确认希望的更新频率、运行星期和本地时间，真正业务歧义时才选择 |
| 4.5 新表初始化 | 仅当你没有现成表时运行 `workbook init-template`，创建周度/日度、无指数/有指数产品页和可见指数来源页；目标已存在就拒绝覆盖 | 无需选模板；打开文件确认工作表名称即可。已有工作簿不会执行这一步 |
| 5. 单次验收 | 先运行 `doctor` 检查本地环境，再由一次 `preview` 邮箱扫描自动完成发现、历史验证和预览 | 听 AI 汇报结果；有拦截按提示解决 |
| 6. ★ 检查预览 | 告知预览文件路径（运行目录的 `previews\` 里） | 有新增时亲自打开预览 Excel，核对原结构、产品标识、日期、净值、收益、基准和样式；零新增时检查中文基线报告，确认邮箱与工作簿最新日期一致、待写入为 0 |
| 7. 批准写入或基线 | 有新增时执行备份 → 临时副本 → 校验 → 原子替换；零新增时只核验并批准当前基线，不改工作簿 | 明确告诉 AI 已检查预览或基线报告并同意启用自动更新；只有实际写表前需要关闭正式表 |
| 8. 自动更新 | 按你在首次配置中确认的频率和时间安装 Windows 计划任务 | 后续无需操作；每次写入前自动备份，异常时原表不变 |

## 日常使用节奏

- **正常情况**：邮件到达后，计划任务自动核对、备份并写入，不需要打开预览。
- **想查看状态**：让 AI 运行 `navctl.py schedule status`，它会报告本地时区、下次运行时间、最近一次是否写入、行数和备份路径。
- **出现异常**：任务停止且不改正式表，让 AI 查看 `logs\update-日期.log` 并修复；手动更新与计划任务都会留下同格式日志，写表配置变化后只需重新验收一次。
- **新增产品**：只需对 AI 说“我已经建好【Sheet 名】，请接管”，或“新增【产品】，格式照【参考 Sheet】”。前者由 `products adopt` 原样接管用户模板，后者由 `products clone` 先备份，再用 Excel/WPS 完整复制格式并清空参考产品业务内容。AI 自动匹配邮箱，不让你填写发件人、代码或内部模式；完成后只检查一次新产品预览，原计划任务继续使用。

## 常见拦截与解决

| 看到的提示 | 原因 | 怎么办 |
|---|---|---|
| 输入授权码时只显示 `*` | 程序用星号遮住每个字符，避免显示授权码正文 | 这是正常现象；右键粘贴后按回车，看到“已加密保存”即完成 |
| 没有出现授权码输入窗口 | AI 没有调用 `secret launch`，或窗口启动失败 | 让 AI 主动重新弹出根目录的 `首次授权.bat` 并检查启动结果；你不需要自己输入终端命令 |
| BAT 一闪而过，或提示“不是内部或外部命令” | Python 未加入 PATH、运行目录混用了不同版本，或路径被 `cmd.exe` 错误解析 | 不要反复双击、移动 `app` 或自己重写 BAT；让 AI 检查三个入口、内部 `.venv` 和控制器，并在真实 `cmd.exe` 中重现。混合版本应新建完整运行目录，不要覆盖修补 |
| 此命令需要用户在真实终端中运行 | AI 错把 `secret set` 放进了管道或后台任务 | 让 AI 改用 `secret launch` 弹出独立窗口，不要自己代输授权码 |
| AI 说“没有安装 Excel/WPS”，但电脑实际装有 | 当前 Python 可能在 WSL/容器中，或软件已安装但 COM 首次启动/注册尚未通过 | 让 AI 检查 `doctor` 的 `runtime_platform`、`spreadsheet_apps_detected` 和具体 COM 错误，主动打开 Excel/WPS 完成首次设置后重试；以虚构 `demo commit` 是否通过为准 |
| AI 说“IMAP 正在冷却”或反复断线 | 网络临时异常，或 AI 重复运行了多次邮箱扫描 | 网络恢复后让 AI 只重试一次；正常流程是成功运行一次 `propose`，配置后直接运行一次 `preview`，不要连续执行 `discover / validate / preview` 三轮扫描；没有服务商明确限频证据时不要把普通断线叫作“冷却” |
| 长扫描中途提示 IMAP 会话断开 | 短暂网络或服务端连接中断；新版会先在同一只读扫描内自动重连两次 | 让 AI 查看 `imap_reconnects` 和进度；UIDVALIDITY 不变时程序从当前 UID 继续，持续断线或邮箱身份变化才完整停止。不要使用不完整结果或手工缓存原始附件 |
| 候选扫描显示 `partial: true` | 当前主题范围达到单批邮件数或软时间预算，不是死锁 | 让 AI 查看 `route-proposal-progress.json` 的阶段、邮件和附件计数，再运行 `propose --resume`；不要重新开始同一范围或使用部分报告接管 |
| Windows 安装目录过长 | 工作簿同级的默认 `净值自动化\` 路径超过 116 字符 | 让 AI 用 `--destination` 换成 `D:\nav-runtime` 这类新的短路径；不要覆盖已有运行目录 |
| Target workbook already exists; refusing to overwrite it | 使用了新建模板入口，但目标路径已经有文件 | 不要删除或覆盖；让 AI 判断这是应改用 `--workbook` 的已有表，还是应另选一个真正不存在的新路径 |
| “正在通过默认 PyPI 安装锁定依赖”长时间无进度或最终失败 | 国内网络直连默认 PyPI 较慢 | 让 AI 在当前引导结束或回滚后，用 `--index-url https://pypi.tuna.tsinghua.edu.cn/simple` 重新部署；它只影响本次安装，不要修改全局 pip 配置 |
| Another runtime process is active | 上一次运行还没结束 | 等几分钟再试；程序崩溃后锁会自动释放，不需要手动删文件 |
| 无法替换正式工作簿……关闭窗口 | 正式表正被打开 | 关闭 Excel/WPS 中的正式表，重新写入 |
| latest routed NAV exceeds max_staleness_days | 该产品邮件长期未更新 | 核实是否赎回/停更；是则让 AI 设 `paused` |
| no routed NAV rows were found | 已配置产品在回看窗口内没有净值邮件 | 让 AI 重新运行 `propose`，检查邮箱文件夹、回看天数或产品是否停更 |
| NAV row was not routed | 邮件中的某行无法唯一匹配产品，或旧运行目录把明确未配置产品也当成错误 | 让 AI 先看脱敏路由报告并确认运行版本；新版会忽略带明确未配置代码的行，没有代码时可用产品名称、专用发件人或主题证明归属，只有仍然歧义才停止。不得另写 `nav_sync.py` 或直接写表任务 |
| 多产品预览运行很久后提示路由重叠或同日冲突 | 多条主题范围同时命中，或同一日期存在不同候选净值 | 让 AI 先读取 `route-report.json` 中的最小重叠路由对、冲突日期和脱敏来源指纹，修正后用 `discover --sheet` 复查路由，或用 `preview --sheet` 查看不可提交的局部工作簿预览；局部通过后必须再跑一次完整预览 |
| 同一产品出现大量估算、交易确认或月报/季报 PDF 错误 | 路由主题过宽，把非正式净值材料收进来了 | AI 应恢复正式净值通知的稳定主题前缀并保留产品代码约束；程序会按脱敏类别报告已排除原因。供应商特有主题核实后加入内部排除片段，不要为了月报/季报新增净值 parser |
| 原邮件之后又收到“更正/以此为准”通知 | 托管人正式修正同一产品、同一净值日的数据 | 程序仅在更正主题明确、邮箱顺序证明后发且单封内部值唯一时采用更正，并在 `corrections_applied` 保留脱敏替代记录；普通重发或晚到邮件不会自动覆盖 |
| 主题有唯一产品代码，但附件净值行没有代码 | 托管附件省略了每行重复产品身份 | AI 只有在发件人和代码主题都精确限定、同封邮件只有一个无冲突序列时，才保存显式主题代码绑定；预览会展示证据。混合代码或宽泛主题仍停止 |
| 邮件头已匹配，但候选产品仍为 0 | 邮件确实无净值，或正文/HTML/附件格式尚不受通用解析器支持 | 让 AI 查看 `parse_gap_summary` 的正文类型、附件 MIME 分布和受控失败类别；需要时用脱敏样例增加本地 parser，不要直接断言邮箱无数据 |
| 中文主题筛选显示 0 封，但邮箱里明明有邮件 | 外部 Agent 或旧终端把中文命令参数按错误代码页传给程序 | 让 AI 改用本地 UTF-8 参数文件入口，并检查报告中的 `filter_input_encoding`；编码核验未通过前不要把 0 解释为真实零命中，也不要另写邮箱扫描脚本 |
| “全部产品”数量对不上 | 有些 Sheet 是参考页、已结束产品、无邮件证据、需本地解析器或业务待确认，并不都应建立活动路由 | 让 AI 查看 `products status` 的 `workbook_coverage`，做到每页有结论；不要为了凑路由数量强行接管 |
| summary row was not found | 路由处于默认严格模式，但工作表数据区后面没有“累计/合计”行 | 如果它本来就是简单追加表，让 AI 核实后显式配置 `sheet_mode: append`；复杂表不要为绕过错误切换模式 |
| 表头下只有“产品代码/名称 + 日期”的预留行，下一行是“累计/合计” | 这是汇总表的首次冷启动结构，不是需要你决定删哪一行 | 只要产品名称或代码一致并找到一条真实邮件，AI 就应生成预览、自动补全预留行并保留“累计/合计”；不要删除汇总行，也不要补造历史净值 |
| 邮件只有工作簿最新日期之后的新数据，提示没有历史重叠 | 邮箱回看期可能没有保留表内旧日期，不能用重叠值做传统核验 | 产品名称或代码任一能唯一对应时，程序应先生成带警告的首次预览；由 AI 和你核对产品、日期和净值，不需要为了凑两个重叠日期而等待下一封邮件 |
| 新建模板提示 bundled template cold start | 新表还没有两条已写入历史 | 一条真实邮件日期已经可以生成第一次预览；认真检查产品、日期和净值即可。第二条历史核实并写入后，程序自动恢复严格验证 |
| 邮件每天都有净值，但原表按周记录 | 邮件频率比 Excel 模板更密集 | AI 应用全部邮件核验，但仍按原表周度规律写入；不得清空原表后把所有日度邮件灌进去，也不得删掉周收益、指数和超额列 |
| 接管已有 Sheet 后提示邮箱旧历史缺少/多出周点 | 邮箱全历史与原表既有取样边界不同，不代表应自动回插 | 默认把写入边界设为原表最后日期的下一天，从表尾继续；收益、累计和基准公式仍沿用原表完整历史。边界前数据保留并只做诊断；你明确要修旧历史时，再检查并批准单独的修复预览 |
| 周度表接管后只有本周邮件，提示历史证据不足 | 当前自然周尚未完成，按原表规则本来就应零新增 | 产品名称或代码与工作簿尾部唯一一致时，程序生成零新增基线并列出暂缓日期；你批准后可上线自动更新，本周不提前写。身份、历史或数值有冲突时仍停止 |
| 已建产品页有基准/超额列，但 AI 暂时找不到可靠来源 | 旧公式能看见，不等于来源已被证明 | 程序先生成不可提交的审查预览，新增行的未确认基准/超额留空；AI 查明来源并重新预览后才允许正式写入。空白预留列不会拦截接管 |
| 工作簿使用自定义基准名称，公开资料也无法唯一映射 | 这是业务身份问题，不是技术解析失败 | 能唯一核实的基准由 AI 自动配置；只有这一页需要你确认业务身份或持续来源，不猜相似指数，也不拖住其他产品 |
| 邮件代码比工作簿代码多份额后缀 | 可能是同一产品的不同份额写法，也可能是另一产品 | 程序只列为 `alias_pending`；AI 核实邮件历史和产品身份后保存完整精确别名，不自动删后缀或模糊接管 |
| historical value conflict | 邮件数据与表内历史对不上 | 让 AI 对照邮件、备份和工作簿定位；只有来源互相冲突、AI 无法证明哪边正确时再请你确认 |
| historical product code column is not constant | Excel 自动填充可能把同一产品代码递增成多个伪代码 | AI 先用正式邮件和工作表身份核实异常行，只在预览副本中展示修复差异；你批准前不得修改正式表 |
| unit/cumulative NAV history has a sequence break | 某行累计净值缺失、违反明确口径，或单位/累计价差只在中间一行异常 | AI 对照正式邮件核对异常日期并生成只改对应单元格的修复预览，保持其他历史、公式和样式不变；批准后仍需备份和 COM 验证 |
| 某发件人邮件解析不了 | 特殊格式 | 让 AI 在运行目录 `app\parsers\` 里写本地解析器，用历史邮件验证后再启用 |

## 夏普比率、最大回撤等专业指标

这套工具不会卡住后续分析。它先负责把“产品名称/代码 + 日期 + 净值”可靠写成基础序列；之后可以让 AI 在未托管的分析工作表或单独文件中计算夏普比率、最大回撤、年化收益、波动率等。

AI 可按通行方法选择单位/累计净值、频率、年化因子、无风险利率和样本区间，并把口径写清楚；只有不同选择会明显改变结果时再问你。不要把自定义公式直接塞进受管汇总行。

## 你只需要记住三件事

1. AI 会主动弹出授权码窗口；你只在窗口中隐藏粘贴一次，不必发到聊天窗口、输入终端命令或反复说明。
2. 第一次配置或写表规则变化时检查一次预览；同一配置之后会自动备份和写入。
3. 出现冲突时程序会停下且不改原表，让 AI 查日志、邮件和备份处理；只有证据无法判定时才需要你选择。
