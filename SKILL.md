# 部署净值邮件自动化

用户已有工作簿时，以该工作簿为版式、公式和视觉样式的唯一依据，绝不套用内置模板、改表头或重排；用户没有工作簿时，才使用内置完全脱敏模板创建新表。邮箱数据、工作簿副本、凭据、发现报告和运行配置必须留在用户本机。

目标工作簿不是完全空白，或邮箱数据频率与表内历史不同，或任何验证步骤失败时，优先阅读 [references/template-preservation-case.md](references/template-preservation-case.md) 中已经验证的处理思路，不得自行简化工作簿；该案例不是对所有外部格式的穷举，证据不同时应继续本地诊断。

## 使用方式：成功路径与自主探索

本 Skill 提供的是经过真实部署和回归测试的默认成功路径，以及对已知问题的可靠处理方案，不是要求 Agent 把所有新情况强行解释成现有案例的封闭决策树。已知场景与文档前提一致时，优先复用内置命令和方案；前提不同、现有方案失败或文档没有覆盖时，Agent 应基于本地邮箱、工作簿、诊断报告和实际运行结果独立提出假设、验证并迭代，不要因为手册没有现成答案就停止，也不要把内部技术选择转交给普通用户。

除“正式交付红线”外，文档中的扫描批次、时间预算、路径、接管边界、历史范围、解析器、列识别和基准来源都是可配置的可靠默认值，不是所有电脑必须满足的隐藏前置条件。真实部署证据表明默认值不适用时，Agent 应调整参数、配置或本地适配层并留下可复核依据；不得把某次 clean-room 的产品数量、邮件格式、列名或耗时写成全局硬规则，也不得为迁就默认值而关闭冲突校验。

探索阶段允许 Agent 自主决定工具和顺序，包括读取必要的本地原始证据、对工作簿副本试验、比较解析方法、编写临时 Python/PowerShell、生成中间报告、在有明确诊断目的时重新执行有边界的邮箱扫描，以及直接选择通用修复或受信任本地解析器。临时材料只放在 `app/diagnostics/`，不得写正式工作簿、安装为计划任务、泄露授权码或进入公开仓库，问题解决后、交付前删除。探索原型不要求复用全部运行时代码；如果原型证明了新格式的正确处理方式，应把持久能力收敛为 `parser: auto` 修复、`app/parsers/` 本地解析器、明确配置或其他可由标准流程复核的实现。

无论采用哪种探索方法，正式交付都必须满足下列结果红线：用户批准首次预览或零新增基线；产品身份、日期、净值和累计口径有本地证据且无冲突；已有工作簿的结构、频率、公式、汇总和样式不被擅改；正式写入继续经过配置/文件哈希、验证、备份、临时副本、Excel/WPS COM 和原子替换；异常时正式表不变；持续运行由受管计划任务执行。替代方法可以改变“怎么找到答案”，不能省略“怎么证明答案安全”。

## 工作原则

- 用户只负责在 AI 主动弹出的可见窗口中隐藏输入邮箱授权码，并检查、批准第一次真实预览。发件人发现、路由配置、表结构识别和后续运行由 AI 完成，不得要求用户手工抄写发件人邮箱、输入终端命令或逐项填写 AI 能从本地证据确定的信息。
- Windows 首次配置必须主动询问用户希望的更新频率、运行星期和本地时间。可根据邮件历史给出一个推荐值，但必须让用户确认；定时自动更新是正式工作流的一部分，不得留空、跳过或描述成可选功能。
- 第一次批准前只读。第一次预览经批准并成功写入后，该配置自动获得后续自动更新权限；若邮箱与工作簿已经同步、没有新增日期，则批准经验证的零新增基线，不写工作簿也能获得自动更新权限。每次后续写入仍先校验、备份并通过临时副本验证，不再要求用户重复检查预览。
- 在选定的历史范围内，只要邮箱能唯一识别目标产品，就主动把真实记录放进预览，不要求用户手工补数据或理解内部模式。新表或明确要求补历史时覆盖可证明的邮箱历史；接管已认可历史的现有表时可默认只续接表尾。邮箱决定真实数据，现有 Excel 决定列、行频率、公式、汇总、基准和样式；不得把“补历史”变成“另做一张表”。
- 产品无法唯一识别、日期或数值冲突、累计净值口径未确认、出现未知列或工作簿结构变化时，必须停止，不得猜测。唯一的同日值替代例外是：同一受管产品和净值日先有原始通知，邮箱中后发通知的主题明确含“更正/修正/以此为准”或等价英文语义，且该更正邮件内部只有一个无冲突值；程序才以更正值进入预览并保留脱敏替代审计。普通晚到、重发、顺序无法证明或更正邮件内部冲突仍停止。已有 `summary` 页只有基准/超额来源暂时无法证明时，不得卡在首次预览之前：`products adopt` 应进入内部审查状态，生成不可提交的预览并把新行相关基准/超额留空；AI 核实来源、重新预览后才允许正式写入。历史不足时，只有显式 `append`、满足严格条件的 `summary` 预留行，或程序生成的内置模板冷启动可以继续；都必须具备产品名称或代码，并人工检查警告与第一次预览。
- 托管人一封邮件包含多个产品时，只选择代码与活动路由精确相等的行。带有明确代码但未配置或已暂停的产品行可以忽略并记录脱敏计数；缺少代码、匹配多个路由、解析失败或同日数值冲突仍必须停止。不得因一个已证明无关的产品扩大受管范围，也不得把无法证明无关的异常静默跳过。
- 真实邮箱、产品和工作簿可以在用户本机用于发现和配置，但不得写入公开 Skill、测试或 Git 提交。用户为解释需求而举出的真实产品名、代码或 Sheet 名也只属于本地证据；公开示例必须改成“示例产品A”“参考产品页”等完全虚构的占位名称。授权码不得出现在聊天、命令参数或日志中。
- 运行目录必须与 Skill 目录分开，避免升级 Skill 时覆盖本地配置或数据。
- 一份工作簿对应一套独立自动化。默认把运行目录创建为工作簿同级的 `净值自动化/`；不要把程序文件散放在工作簿所在目录，也不要让不同工作簿共用同一套配置、密钥、日志或计划任务。只有默认路径过长、不可写或名称冲突时，才用 `--destination` 选择另一个短本地目录。
- 交付给普通用户的运行目录必须保持简洁。根目录只保留中文说明、可双击入口、保护本地资料的 `.gitignore`、`previews/`、`backups/`、`logs/` 和内部 `app/`；Python 源码、隔离环境、配置、状态、本地解析器和定时任务脚本全部放进 `app/`。不得把调试脚本、临时导出、检查结果或新增 `.py/.ps1/.json` 文件堆在用户根目录；临时诊断放进 `app/diagnostics/`，交付前删除。

## 创建本地运行目录

获取已有工作簿路径；如果用户没有工作簿，获取希望创建的新 `.xlsx` 完整路径。再获取收件邮箱账号。引导程序默认在工作簿旁创建独立的 `净值自动化/`；IMAP 主机优先从现有邮件客户端设置中读取，不足时再查 [references/email-providers.md](references/email-providers.md) 或服务商官方文档。只有确实无法确定账号认证方式时才询问用户。

Windows 上不得假定 `python` 已加入 PATH。AI 先在当前原生 Windows 终端定位可用解释器：优先复用自身已验证的 Python；否则检查 `py.exe -3`，再检查 `python.exe`，并用解析出的实际命令运行引导程序。下面示例中的 `python` 代表该已解析解释器。若两者都不可用，明确报告缺少 Python 并协助安装；不得让普通用户反复输入命令、另写临时 BAT，或把“不是内部或外部命令”解释成电脑不支持。

确认后在 Skill 根目录运行：

```powershell
python scripts/bootstrap.py --workbook "D:\data\nav.xlsx" --email "user@example.com" --imap-host "imap.example.com"
```

没有现成工作簿时，登记一个尚不存在的新路径；此命令只创建运行目录，不会提前生成空壳工作簿：

```powershell
python scripts/bootstrap.py --new-workbook "D:\data\nav.xlsx" --email "user@example.com" --imap-host "imap.example.com"
```

`--workbook` 与 `--new-workbook` 必须二选一。前者要求文件已经存在；后者要求目标尚不存在且只能是 `.xlsx`，任何已有文件都会被拒绝覆盖。省略 `--destination` 时，运行目录固定推导为工作簿同级的 `净值自动化/`；若该目录已经存在，引导程序失败关闭，不会复用或覆盖。只有路径预算或目录冲突确实不允许默认位置时，才显式传入新的 `--destination`。

引导程序会创建隔离的虚拟环境、安装锁定版本的依赖，并生成本地配置。

在中国大陆网络，或直连默认 PyPI 已明显缓慢、超时后，AI 应只为本次部署启用清华镜像：

```powershell
python scripts/bootstrap.py --workbook "D:\data\nav.xlsx" --email "user@example.com" --imap-host "imap.example.com" --index-url "https://pypi.tuna.tsinghua.edu.cn/simple"
```

`--index-url` 只传给本次运行目录的依赖安装，不会修改用户的 pip 配置。不得要求用户运行 `pip config set global.index-url`，也不得静默改变用户其他 Python 项目的软件源；网络正常时省略此参数。

Windows 优先使用工作簿同级的 `净值自动化/`，让工作簿与其配置、入口、备份和日志形成一套清晰的本地交付。引导程序把 116 个字符作为当前可靠支持上限，为内部 `app/` 和深层依赖保留路径预算；默认位置过长、不可写或已有同名目录时，改用 `--destination "D:\nav-runtime"` 一类的新短本地目录，不得绕过预检、覆盖旧运行目录或要求用户修改公司电脑的组策略。

引导完成后必须按照 [references/runtime-layout.md](references/runtime-layout.md) 检查目录分层：用户根目录应包含 `使用说明.txt`、`首次授权.bat`、`查看状态.bat`、`手动更新.bat`、`.gitignore`、`previews/`、`backups/`、`logs/` 和 `app/`。普通用户只接触中文入口和三个结果目录；`.gitignore` 只用于防止本地资料被误提交，`app/` 由 AI 管理，不要求用户进入、理解或修改。后续所有命令都在 `app/` 中执行：

三个用户 BAT 必须由引导程序生成，使用运行目录内的固定 Python、ASCII 内容和 CRLF 换行；不得让 Agent 重写或部分覆盖。交付前，AI 必须在真实 `cmd.exe` 中分别启动三个入口，确认错误会显示并留窗、不会出现“不是内部或外部命令”，并覆盖包含中文、空格、`&` 和括号的安全临时路径。详细反例见 [references/runtime-layout.md](references/runtime-layout.md) 的“BAT 闪退与命令无法识别”。`手动更新.bat` 与计划任务必须共同调用内部 `app/run-update.cmd`，让首次尚未批准、配置错误等写入前失败也生成同样的 `logs/update-YYYYMMDD.log`；用户入口负责留窗，内部入口负责日志。

AI 在后台运行一次完全离线的虚构演练，不要求用户参与：

```powershell
cd D:\data\净值自动化\app
.\.venv\Scripts\python.exe navctl.py demo prepare
```

该命令不读取真实配置、密钥或工作簿。Windows 上继续验证只作用于虚构工作簿的 COM 写入：

```powershell
.\.venv\Scripts\python.exe navctl.py demo commit --run-id <run_id> --yes-reviewed-preview
.\.venv\Scripts\python.exe navctl.py demo remove --run-id <run_id>
```

演练通过只证明目标电脑和通用流程可用。

Windows 上必须由 AI 主动弹出独立、可见的授权码窗口：

```powershell
cd D:\data\净值自动化\app
.\.venv\Scripts\python.exe navctl.py secret launch
```

不得只把 `secret set` 命令发给用户，也不得让用户自己打开 PowerShell、切换目录或复制长命令。`secret launch` 返回 `launched: true` 后，立即明确告诉用户：“已打开授权码窗口，请在新窗口右键粘贴授权码并回车；屏幕只显示星号。”用户只负责粘贴和回车，AI 不得代输、通过管道传入或查看授权码。

用户操作后，AI 必须自行运行以下命令确认成功；`available: true` 之前不得连接邮箱或声称部署已就绪：

```powershell
.\.venv\Scripts\python.exe navctl.py secret status
```

如果没有出现窗口，AI 先检查 `secret launch` 的结构化错误；必要时从运行目录根目录主动用 `Start-Process -FilePath ".\首次授权.bat" -WorkingDirectory (Get-Location) -WindowStyle Normal` 重开可见窗口。只有自动弹窗确实失败时才向用户说明具体阻碍，不得反复要求用户自己输入命令。用户按 Ctrl+C 取消时不会保存密钥。

在 macOS/Linux 上，授权码只保留在当前 shell；运行程序不会持久化明文密钥：

```bash
cd /opt/nav-runtime/app
read -rsp "IMAP authorization code: " NAV_EMAIL_PASSWORD && export NAV_EMAIL_PASSWORD && printf '\n'
```

当前版本支持通过 SSL 连接 IMAP，并使用应用专用密码或授权码；对 163、126、yeah.net 和网易企业邮主机会在登录后、选择邮箱前发送不含账号信息的 IMAP ID。仅限 OAuth 的邮箱登录暂不支持；即使服务器地址正确，也必须先核对账号认证方式。PDF 仅解析文本，不提供 OCR。

## 自动发现并配置路由

授权码保存后，邮箱范围较大、发件人未知或需要最小化首次读取时，先由 AI 只读取最近一批邮件的 `From/Subject`：

```powershell
.\.venv\Scripts\python.exe navctl.py propose --headers-only --header-limit 25
```

该命令不读取正文、附件或邮件大小，报告只写入本地 `app/route-proposal-headers.json`。AI 用工作簿中的产品名称/代码与头部报告确定候选发件人和稳定主题片段；样本不足时可以有目的地把 `--header-limit` 扩到 100 等合理范围。完整解析必须至少限定精确发件人；主题稳定时再一并限定：

```powershell
.\.venv\Scripts\python.exe navctl.py propose --sender "sender@example.com" --subject-contains "稳定主题片段"
```

在外部 Agent、旧版 `cmd.exe` 或其他可能改写中文命令参数的 Windows 终端中，不要用诊断脚本的“零命中”直接断言邮箱无邮件。把完整参数按“一行一个参数”写入 UTF-8（可带 BOM）文件，再运行 `navctl.py @diagnostics\propose-args.txt`；文件中不加 shell 引号，也不支持嵌套 `@` 文件。CLI 会拒绝非 UTF-8 文件、替换字符和可识别的错误解码，并在成功报告的 `scan.filter_input_encoding` 标明 `utf8-argfile` 或 `native-unicode-argv`。参数文件只属于本地诊断资料，确认后清理，不得进入公开 Skill。优先使用 ASCII 产品代码和精确发件人并不等于禁止中文主题；需要中文时使用该入口，不得另写邮箱扫描器。

选择式 `propose` 会先让 IMAP 服务端按精确发件人缩小回看期结果，再只读取该范围的最小邮件头；主题仍在本地精确复核，只有同时命中的邮件才查询大小并下载完整 MIME。服务端筛选只是性能优化，程序仍必须重新解析并核对真实 `From`，不能把服务端搜索当成身份认证。扫描报告应区分整个回看期、服务端发件人范围、已读头部和完整下载数量。真实发件人和主题只出现在本地命令与报告中，不让用户手工抄写。若产品身份只存在于正文或附件、主题无法可靠判断，仍然使用发件人范围，不省略 `--sender`：

```powershell
.\.venv\Scripts\python.exe navctl.py propose --sender "sender@example.com"
```

首次选择式解析默认只回看不超过 30 天、每批最多 50 封，并为下载与解析阶段分别使用 120 秒软时间预算。CLI 会持续输出 `search/headers/sizes/messages/parse` 阶段、当前邮件计数和已解析附件数，同时把不含真实发件人或主题的状态写入 `app/route-proposal-progress.json`。范围未完成时不会发布可接管的正式候选，而是写入 `app/route-proposals.partial.json` 和稳定 UID 游标；重复运行 `propose --resume`，直到返回 `partial: false`。不得因仍有 CPU 活动就无限等待，也不得把部分报告用于接管。

读取过程中若 IMAP 会话短暂断开，运行时在同一次只读扫描中最多自动重连两次，并重新执行当前无副作用的搜索、邮件头/大小批次或当前 UID 正文读取。只有重新连接后的 `UIDVALIDITY` 与初始邮箱一致才继续；报告中的 `imap_reconnects` 和进度阶段 `reconnect` 记录次数。持续断线、重新登录失败或 UIDVALIDITY 变化仍失败关闭，不发布不完整发现结果。不得把原始邮件或附件长期缓存到 Skill/运行目录来绕过校验；选择式 `propose` 使用既有 UID 游标恢复，完整 discovery 成功后才使用受指纹保护的解析结果快照。

优先用“单一产品的稳定主题 + 短窗口”完成第一轮身份和工作表匹配，再按同一产品逐步扩大窗口；主题本身稳定时，IMAP 服务端同时按发件人和主题缩小结果。需要补足更早历史时对候选显式增加 `--lookback-days`，仍按游标分批恢复；路由确认后用同样范围运行一次 `preview --lookback-days ...`，该临时范围进入本次历史方案但不放大日常计划任务窗口。若单封复杂附件超过软预算，当前附件结束后暂停并保留游标；Agent 应查看进度和脱敏解析警告，不要把它误判为死锁。

只有扩大有界头部样本后仍无法确定任何候选发件人，且 Agent 已检查 `lookback_days`、`max_messages`、单封及总字节上限并有明确诊断理由时，才可使用 `propose --allow-unscoped-full-scan`。普通无参数 `propose` 会在连接邮箱前拒绝，防止把一次昂贵的长窗口完整扫描误当默认步骤。这个显式出口用于未覆盖情形，不得在刚生成头部报告后顺手使用；超时后也不得立即重复同一全范围调用。

完整候选报告写入本地 `app/route-proposals.json`。AI 读取该报告；已有工作簿时再读取工作簿，自动完成“发件人/产品 → 工作表”、列映射、累计净值、收益频率、序列起点和基准。不要让用户手工提供发件人邮箱；只有出现多个同样合理的映射或业务口径无法从历史证明时才询问。

若主题中的产品代码与工作簿唯一一致，而正文或附件净值行不重复代码，Agent 可在已经限定同一发件人和精确代码主题时增加 `--subject-product-code "示例代码A"`。该值必须与 `--subject-contains` 标准化后完全相等；同封邮件解析结果必须全为无代码的单一无冲突序列。报告会保存 `exact-sender-and-subject-code` 证据和绑定邮件数，接管后写入路由的 `subject_product_code`，预览也会显示使用次数。出现混合代码、同日不同值、多个适用路由或主题不是唯一代码时继续停止。

若邮件头已经匹配但通用解析器最终得到 0 条净值，`parse_gap_summary` 必须给出匹配邮件数、纯文本/HTML 分布、附件 MIME 类型计数、首个受控失败类别和 `local_parser_recommended`；不得记录正文、完整主题、附件名或异常原文。Agent 据此判断应扩展通用解析器还是在 `app/parsers/` 增加本地解析器，不得把“0 候选”误报成邮箱没有数据。

一次成功且仍与当前假设一致的完整 `propose` 结果应直接用于配置，不必为了形式上的“再确认一次”立即重复扫描。头部预筛选与其后一次选择式完整解析属于同一轮候选发现，不是重复扫描。若输入、配置或诊断假设发生变化，Agent 可以有目的地重新扫描，但应复用已有报告、遵守数量/字节边界并避免连续快速登录。除非服务商返回了明确的限频证据，不得自行宣称邮箱正在“冷却”。

路由已经配置后，共享托管发件人的邮件应先由 IMAP 服务端按各活动路由的发件人和稳定 `subject_contains` 取并集，再在本地批量读取最小邮件头并精确复核，最后才计算 `max_messages`、读取大小和下载正文/附件。`max_header_messages` 约束服务端已缩小的邮件头集合，超限时要求缩短回看窗口或使用更精确主题，不能静默截断。没有配置主题过滤的路由仍视为授权该发件人的全部范围，不能静默跳过无法解析的邮件；但邮件成功解析后，带明确代码且不属于任何活动路由的行应由程序选择式忽略，不得因此要求用户接管其他产品。

优先把每条路由的 `subject_contains` 收窄到托管人正式净值通知的稳定前缀，并继续保留产品代码约束。核心运行时在下载正文前排除主题明确属于虚拟/模拟净值估算，以及不含明确净值通知语义的交易确认、月报或季报，`excluded_non_nav_reasons` 只按受控类别计数；同时含明确净值语义时继续解析，不得仅凭“月报/季报”静默跳过。供应商特有的非净值主题由 Agent 查看本地原始证据后写入内部 `subject_excludes`；不得让用户填写，也不得用宽泛词排除可能的正式净值通知。报告把“非净值通知”与“解析器缺口”分开；月报/季报 PDF 不应因为附件存在就误补 PDF 解析器。

生成首次配置时，根据邮件到达记录向用户推荐运行安排，并明确询问：每周一次、工作日每天或哪些星期运行，以及目标电脑本地时间。把用户确认的日期和时间写入 `schedule`；不得自行决定，也不得以空数组完成 Windows 部署。

编辑 `app/config.json` 前完整阅读 [references/configuration.md](references/configuration.md)。

- 常见的带标签纯文本、HTML 正文表格以及 Excel/CSV/PDF 表格优先使用 `parser: auto`。
- 通用解析器将“净值日期”“估值日期”“估值基准日”“业务日期”等识别为净值日期。新增别名必须使用完全脱敏样例回归，并保持日期列唯一识别；不得为了通过单个附件而放宽到含义不明确的字段。
- 产品确认赎回、清盘或长期停更时，设置 `paused: true` 和明确的 `pause_reason`；不得用暂停掩盖活动产品的意外解析失败或陈旧数据。
- 同一发件人可能发送多个产品时，优先用产品代码分流；没有稳定代码时，可用产品名称和互不重叠的 `subject_contains` 形成唯一路由。只有多条路由仍会同时命中时才停止。
- 同一发件人的不同产品使用不同解析器时，优先用互不重叠的 `subject_contains` 分流。
- 已有工作簿根据实际结构在 `summary` 与 `append` 中自动选择：`summary` 用于带固定汇总行和受管公式的复杂表，`append` 用于空白或无汇总行的纯追加表。模式由 AI 检查工作表后决定，不得要求用户选择，也不得为绕过验证而切换。`template` 仅供程序生成的新工作簿使用，禁止套到已有文件。
- 如果结构恰好是“表头 → 一条只有产品名称/代码和日期、净值为空的预留数据行 → 紧邻的累计/合计行”，应自动按 `summary` 预留行冷启动处理，不得询问用户删除汇总行，也不得要求或制造一条虚假历史净值。预留行身份与路由精确一致、邮件至少包含一个真实日期且该行没有其他业务内容时，用最早一封真实邮件替换预留日期并补全净值，再把其余历史数据插到汇总行之前；汇总行和受管公式必须保留，第一次预览仍由用户检查。
- 首次 `summary` 预览的目标是把可审查结果跑出来，不是提前达到无人值守所需的最高证据强度。工作簿中的产品名称或产品代码任一与唯一邮件路由一致，即可在只有一条真实邮件、零历史重叠或只有一个重叠日期时生成带警告的预览；不强制要求用户提供代码，也不要求主题中重复出现代码。邮件中的日期只能是工作簿已有日期或尾部新日期，已有同日值必须一致。产品身份明确冲突、多条路由同样可能、历史内部插入、累计口径或异常跳变仍停止。Agent 必须检查报告和预览，用户批准前不得写正式表。
- 邮件比模板更密集时，所有邮件都可参与身份和数值核验，但写入频率必须沿用模板。已有表保持 `data_frequency: auto`，让运行时优先从现有日期规律、其次从明确表头判断。新建模板由 AI 根据邮件日期规律明确设为 `daily` 或 `weekly`，并让 `return_frequency` 与之相同；只有证据冲突才问用户。周度表只写每个已完成自然周最后一条可用净值；若接管后只有当前未完成自然周的真实邮件，产品名称或代码与工作簿尾部已经唯一核实，则生成可批准的零新增基线并在报告中标明暂缓日期，不得误报证据不足，也不得提前写入本周。日度表维护日收益，并只在每个已完成自然周最后一个可用日期维护周收益。计划任务运行频率与数据频率不是一回事。
- 历史校验必须区分接管边界前的保留区间与边界后的受管区间：旧异常默认只诊断、不改写，受管区间仍检查身份、累计序列和公式链。完整边界语义与历史修复流程见 [references/configuration.md](references/configuration.md)。
- 优先用语义表头和已确认的基准显示名识别列；具体指数名称与相邻“该名称（日收益）”可作为基准点位/收益配对。`products adopt --inspect-only` 必须展示推断出的列角色。语义仍不充分时，由 Agent 使用显式列映射，不把本机指数名写进通用规则。
- 每条路由都要设置累计净值策略。除非历史证据证明应使用 `unit` 或固定 `offset`，否则保持 `require`。
- 接管已有历史的 Sheet 时，Agent 根据用户目标和本地证据选择历史范围：普通续接默认只允许从原表尾部新增，但收益、累计和基准公式继续沿用原表完整历史；需要补历史时可选择受监督的邮箱历史范围或设置自定义 `series_start`。这属于内部配置，不让普通用户选模式；任何历史新增或修复仍进入首次受保护预览。完整字段见 [references/configuration.md](references/configuration.md)。
- 只有可靠资料明确证明产品基准时才配置指数；否则使用无指数模板。基准按精确日期对齐。已有表映射到已核实来源工作表；新建表使用可见的指数数据页并配置 `display_name`，把“基准指数”替换为“中证1000”等已核实名称。需要公共指数源时完整阅读 [references/index-data-sources.md](references/index-data-sources.md)，核验代码、口径、历史重叠和使用条款后再写入本地来源页，不得猜测。

使用 `--new-workbook` 时，在路由、日/周频率和基准全部写入 `app/config.json` 后运行：

```powershell
.\.venv\Scripts\python.exe navctl.py workbook init-template
```

程序按“一只产品一个工作表”创建所需页面，把共享指数来源合并为工作簿末尾的可见数据页，并保留空白产品说明行。目标文件已存在、路由未确认、频率仍为 `auto` 或工作表名冲突时必须停止；不得删除目标后重试或覆盖用户内容。

## 普通用户新增产品的两种自然语言入口

普通用户只需要用自然语言说明“我已经建好一个 Sheet”或“新产品照哪个 Sheet 的格式”。这两种入口约束的是用户需要提供的信息，不限制 Agent 为兼容特殊工作簿而采用的本地诊断和实现方法。不得让用户选择 `summary/append`、填写发件人、抄产品代码或编辑 JSON。AI 用邮箱候选、产品代码、工作簿表头和历史日期唯一确定 `proposal_index`；刚完成选择式 `propose` 时运行 `products sync --use-existing-proposals`，不要重复登录邮箱。确需刷新时运行 `products sync --sender ...`，主题稳定时再加 `--subject-contains ...`；无范围刷新同样必须显式使用受控诊断开关。

`products status` 和 `products sync` 的 `workbook_coverage` 必须覆盖工作簿每一个 Sheet。“全部产品”指逐页已有可审计结论，不等于每页都应建立活动路由。活动路由、基准待审路由、暂停路由和基准参考页由程序自动分类；其余页由 Agent 根据本地工作簿、邮件头/候选报告和业务证据写入内部 `sheet_reviews`，状态只能是 `excluded`、`no_mail_evidence`、`local_parser_required` 或 `business_review`，并附一条本地理由。`all_sheets_classified: true` 只证明盘点闭环；`action_required_sheets` 仍会计入基准待审、解析器待补、业务待确认和未分类页。不得为凑活动路由数量接管虚拟净值、报告/合同、已结束产品或无证据页面，也不得把真正歧义标成“排除”。这只是 Agent 维护的覆盖审计，不增加普通用户需要填写的产品管理选项。

### 路径一：接管用户已建好的 Sheet

适用于用户已经在 Excel/WPS 中建立产品页的情况。只要表头能唯一识别日期和单位净值，并且数据行含产品名称或代码及一个真实日期，AI 直接运行：

```powershell
.\.venv\Scripts\python.exe navctl.py products adopt --proposal-index 1 --sheet "用户已建产品页"
```

工作簿含多只产品时，AI 先对候选逐一添加 `--inspect-only` 做只读接管检查，不修改配置或工作簿。把 `ready_for_direct_adoption: true` 的产品先接管并完成第一份可提交预览；`review_required: true` 的基准/超额页留在单独审查队列，查明来源后再接管。不得先把全部候选写入配置，导致少数基准歧义产品阻塞其他证据完整产品上线；也不得用 `paused` 掩盖待审查状态。

`products adopt` 自动从工作表判断纯追加/带累计行、日频/周频、产品代码、产品名称、累计净值口径、列角色和建议接管边界；工作簿字节保持不变。已有历史页默认采用续接范围；若用户要求补历史或本地证据表明应复核旧区间，Agent 内部改用 `--history-scope mail-history`，让邮箱历史进入首次受保护预览。预留行或空表保持冷启动。只有一个邮箱候选仍含多个代码或累计净值口径无法由邮件与历史证明时，AI 才传 `--code`、`--product-name` 或询问用户。不得因为内部模式不确定而把选择题交给用户。

如果目标 `summary` 页已有非空基准点位、基准收益或超额公式，但可靠来源暂时无法从本地证据证明，`adopt` 会自动保存内部 `benchmark_review_only` 状态，不要求用户填写配置。继续运行 `preview`：它保留净值和样式，在待新增行及汇总行把未确认的基准/超额单元格留空，并在 `plan.json` 标记 `committable: false`。这份预览用于让 Agent 找出来源和验证映射，不能执行 `commit` 或安装自动更新；AI 配置已核实的 `benchmark`、移除审查状态并重新预览后，才进入普通批准流程。权威来源与工作簿基准可唯一对应时自动完成，不必询问；工作簿使用自定义名称且本地与公开证据都不能唯一映射到持续数据源时，才把这一页保留为真正的业务歧义并请用户确认，不得猜相似指数。多产品部署继续让其余证据完整产品先预览。若相关列只是空白预留列，不触发审查状态。

若邮箱代码只比工作簿主代码多一个份额级别后缀，`products sync` 只把它列入 `alias_pending`，绝不做模糊接管。Agent 用邮件历史、产品名称和 Sheet 身份核实为同一产品后，运行 `products alias --sheet "用户已建产品页" --code "已核实的完整邮件代码"` 保存精确 `code_aliases`；别名必须出现在当前同发件人候选报告中，且会使首次预览批准失效。后续仍按主代码或完整别名精确相等路由，不执行前缀、包含或自动去后缀匹配。

### 路径二：照现有 Sheet 的格式新建

适用于用户说“新增示例产品B，格式和参考周度产品页一样”。参考页必须是当前配置中已经受管、带累计/合计行的产品 Sheet：

```powershell
.\.venv\Scripts\python.exe navctl.py products clone --proposal-index 1 --sheet "示例产品B" --copy-from "参考周度产品页" --product-name "邮件中的完整产品名称"
```

`products clone` 不是只复制表头：它先备份正式工作簿，在同目录临时副本中通过 Excel/WPS COM 复制整张参考页，保留列宽、行高、表头、字体、边框、条件格式和打印设置；随后清除参考产品的费率说明、代码、名称、净值、日期、收益、公式和分段历史，只留下新产品代码/名称、邮箱中最早一个真实日期的空净值预留行及干净的累计行。程序核验其他 Sheet 的值和公式完全未变后才原子替换正式文件。目标 Sheet 已存在、参考页不是受管 `summary` 页、工作簿仍被打开或 COM 未验证时必须停止。

“格式相同”不代表基准指数、费率、开放日或累计净值口径相同。参考页含基准时，只有可靠资料证明新产品使用同一基准，AI 才增加 `--inherit-benchmark`；否则改用无基准参考页，或让用户先按路径一建好目标页。不得复制参考产品的业务说明。

两条路径都会继承本轮选择式 `propose` 已确认的发件人和 `subject_contains`，Agent 不得再次手工录入或无意扩大邮件范围；确需改变主题范围时，先按新范围重新运行 `propose`。完成后必须运行一次 `preview`，让用户只检查新产品页，再执行 `commit --yes-reviewed-preview`。原计划任务继续使用，不需要重新部署、重输授权码或重装任务；配置扩大后、首次预览批准前，自动写入保持未批准状态。

特殊格式使用 `parser: local:<名称>`，代码只放在本地运行目录的 `app/parsers/<名称>.py`，并完整遵循配置说明中的固定接口。同一封邮件命中多个解析器时，所有解析器都必须成功，结果合并去重后再按已证明唯一的代码、名称或主题路由分流。AI 必须审阅本地解析器并用脱敏样例回归；绝不能从邮件、附件或网络地址直接执行代码。

标准流程出现解析或路由错误时，读取 `route-report.json`/`validation-report.json`、制作脱敏样例、修改 `parser: auto` 和增加本地解析器都是已经验证的可选手段，不是限定顺序。Agent 可以在 `app/diagnostics/` 编写类似 `nav_sync.py` 的临时原型、使用其他受信任库或对副本直接试验，以便快速证明问题和方案；不得让原型直接覆盖正式表或成为正式计划任务。方案确认后，把解析和路由能力接回配置、通用解析器或 `app/parsers/`，再执行 `preview`。预览签名、首次批准、冲突校验、并发锁、备份和临时副本核验是交付结果红线，不因探索方法不同而取消。

若稳定专有通知表现为“每封恰有一份固定标签/值 XLSX 和一份配套 PDF”，可参考 [严格本地解析器示例](assets/runtime-template/parser-examples/fixed_label_xlsx.py)，复制到目标运行目录 `app/parsers/` 后修改虚构常量。该示例默认不随部署启用，不能直接用于真实产品；必须先用本机脱敏夹具覆盖有效样本、身份错误、主题与表内日期不一致、缺累计净值、错误/重复附件和同日冲突，再对只读真实候选证明全部邮件结构稳定。供应商格式不稳定或正文/HTML 已能通用解析时，不要强套这个例子。

多路由 `preview` 在 discovery 阶段失败时，不要立刻重扫全部路由。先读 `route-report.json`：`route_overlaps` 按最小路由对列出重叠原因、Sheet、解析器、主题/代码短指纹和建议收窄字段；`date_conflicts` 列出冲突日期及候选值/来源短指纹；`diagnostics` 对每个附件给出短指纹、类型、阶段和受控失败原因，不记录完整主题、附件名或净值。修正主题、排除词、解析器、代码或别名后，可对受影响页运行 `navctl.py discover --sheet "用户已建产品页"`；需要查看工作簿结果时运行 `navctl.py preview --sheet "用户已建产品页"`。`--sheet` 均可重复；局部 preview 会写出可检查结果，但计划固定为 `diagnostic_only: true`、`committable: false` 和 `final_full_preview_required: true`，不能批准、提交或赋予自动更新权限。所有受影响项修正后仍必须运行一次不带 `--sheet` 的完整 `preview`，证明跨路由没有重叠并生成统一首次验收结果。

同日不同值先检查是否属于正式更正通知，不要直接选择最新邮件。核心运行时只在路由和日期相同、明确更正主题后发、所有候选都有可比邮箱顺序且单封更正内部唯一时应用替代；`corrections_applied` 保存更正标记类别、顺序依据、替代与被替代消息/主题/数值/来源短指纹。完整主题、原始净值和附件名不得进入报告。Agent 应在第一次预览中核对该审计；没有 `corrections_applied` 的普通冲突继续根据 `date_conflicts` 诊断，不得用本地“最后一封赢”规则扩大到所有邮件。

工作簿历史若被邮件证据证明存在错误，不得因为证据充分就直接修正式表，也不得把错误行静默当作已验证历史。先按 `history_integrity` 将范围缩到具体行和日期，再用正式通知逐项核实；只在 `previews/` 中生成保持其他单元格完全不变的历史修复副本和差异清单，列出 Sheet、日期、单元格、修改前后短指纹及邮件证据短指纹。用户必须像首次预览一样检查并明确批准。批准后才能沿用备份、同目录临时副本、Excel/WPS COM、修复后验证和原子替换保护应用；当前运行时若还没有能表达该修复的受保护计划，就停在修复预览并扩展标准工具，绝不能手工改正式表。修复完成后重新运行一次完整 `preview`，原首次自动更新批准不得沿用。

## 首次验证和一次性预览

先运行本地环境检查，再用一次邮箱会话完成发现、历史验证和预览：

```powershell
.\.venv\Scripts\python.exe navctl.py doctor
.\.venv\Scripts\python.exe navctl.py preview
```

`preview` 内部已经依次执行邮件发现和历史验证，并在 `app/` 写出 `route-report.json`、`validation-report.json`，在根目录 `previews/` 写出预览结果；有待写入日期时生成预览工作簿，没有新增日期时生成中文零新增基线验收报告。默认首次部署直接运行一次 `preview`，不为形式完整而预先串行重复 `discover` 和 `validate`。定位具体阶段、比较新方案或验证发生变化的假设时，可以有目的地单独运行这些诊断命令。

一次完整 discovery 成功后会在 `app/` 保存受指纹保护的本地发现快照。后续只调整列映射、基准、公式兼容或其他工作簿侧规则时，运行 `preview --reuse-discovery`：程序先重新读取轻量邮件头、UID 和大小，只有邮箱范围、路由选择、解析器源码和接管边界全部未变才复用已解析净值；任一项变化都拒绝复用并要求普通 `preview`。该方式仍重新执行当前工作簿验证、生成新的受签名完整预览，不是跳过最终验收，也不要求普通用户理解缓存。

在 macOS/Linux 上，在运行目录的 `app/` 中把 `.\.venv\Scripts\python.exe` 换成 `.venv/bin/python`。这些平台只支持发现、验证和预览；正式写入仍仅支持 Windows。

只有同时满足以下条件，才能接受配置：

1. 每个受管工作表的日期列和单位净值列都能唯一识别；显式追加模式的空白工作表会在预览中初始化标准表头，内置模板必须由 `workbook init-template` 创建。
2. `summary` 模式的数据区后必须紧跟可识别的汇总行；汇总行存在非程序管理公式时必须停止。`append` 模式不要求汇总行，但数据区下方不得有页脚内容。
3. `minimum_history_dates` 是无警告历史核验目标，不是首次预览的启动门槛。普通 `summary`、严格识别出的预留行、`append` 和程序生成的 `template` 都可凭一个唯一的产品名称或产品代码及至少一个真实邮件日期生成带警告的首次预览；达到两个已写入且核实的历史日期后恢复普通严格验证。不得伪造日期凑门槛，也不得把产品冲突或历史内部插入当成普通冷启动。
4. 用于身份确认的产品名称或产品代码至少有一个一致；如果已有日期重叠，单位净值及邮件提供的累计净值必须在配置容差内一致。
5. 新的尾部日期只建议写入一次且顺序正确；不得存在重复日期或同日数值冲突。发现历史内部缺口时先停止写入，由 AI 对照邮件、备份和工作簿诊断并提出修复；只有来源相互冲突、无法证明哪个值正确时才询问用户。
6. 日收益使用上一个有效日期；周收益只出现在已完成自然周的最后一个可用日期。
7. 基准收益与超额收益必须同时有值或同时留空；来源待审时两者都留空，预览标记为不可提交，不能为了跑通而翻译或猜测旧公式。
8. 可提交预览保留工作簿结构并按模板频率包含应写入的历史记录；原有工作表、历史值、公式列、汇总行、基准列和样式不得减少，并通过内置公式和幂等性回归测试。审查专用预览只允许清空计划明确列出的待确认基准/超额单元格，正式表仍完全不变。

如果没有新增日期，`preview` 不复制一份完全相同的工作簿，而是在 `previews/` 生成 `.txt` 基线报告并保留受哈希保护的 `app/plan.json`。AI 必须向用户说明“当前无数据需要写入，但配置仍需首次批准”，让用户检查报告后继续执行同一条 `commit --yes-reviewed-preview`；该命令只核验报告、配置和正式表哈希，记录自动更新批准，不创建备份、不启动 COM、不修改工作簿。完整错误反例见 [references/configuration.md](references/configuration.md) 的“零新增首次验收与错误反例”。

`doctor` 会分别报告 `bootstrap_ready`、`mail_discovery_ready`、`preview_ready`、`commit_ready` 和 `schedule_ready`。存在 `benchmark_review_only` 时 `preview_ready` 可以为真，便于先跑出结果；`write-rules-resolved`、`commit_ready` 和 `schedule_ready` 必须为假，直到 AI 解决来源并重新预览。

有新增数据时，预览是本地工作簿副本。已有表必须与原表并排检查：工作表列表、原有历史、日期频率、表头、公式数量与范围、汇总行、基准列和格式都应保留。普通公式按文本核验；不受插行影响的数组公式按公式文本和作用范围核验，不能因 openpyxl 对象身份不同误报改变。数据区数组、多单元格数组以及任何可提交预览中会被移动、跨越插入点或位于模板复制行的数组公式必须停止，因为 openpyxl 不能可靠平移其作用范围。唯一审查例外是：`benchmark_review_only` 的不可提交预览中，单单元格数组恰好位于受管汇总行的基准点位/收益/超额单元格；程序可在副本中受控清空，并在 `review_array_formulas_cleared` 记录坐标，正式表不变。来源解决后最终可提交预览不得沿用此例外；先通过 Excel/WPS COM 或可证明等价的方法重建普通/数组公式并回验公式范围与计算结果，再重新完整预览。新建表重点检查产品标识、日/周频率、是否应有基准、首批真实日期、净值、日/周收益和超额公式，以及等线 10 号、浅蓝表头、收益浅黄、超额亮黄、红涨绿跌。审查专用预览还要检查 `blocking_reviews`，不得把它当作已批准结果；解析 Excel 附件时若 openpyxl 发出修复/兼容警告，报告只记录脱敏的库名、来源类型、警告类别和数量，Agent 必须核对后再批准。零新增时检查基线报告中的产品页、频率、已核验历史数量、邮箱与工作簿最新日期及待写入数量。不要直接修改正式工作簿。

夏普比率、最大回撤、年化收益等派生指标不由 navctl 直接计算。AI 可基于经验证的净值或收益序列，在未托管分析工作表或单独文件中按通行口径计算并说明假设；只有不同口径会实质改变结论时才询问用户。不要把自定义分析公式放进受管汇总行。

## 通过 Excel 或 WPS 正式写入

正式写入要求 Windows，以及可用的 Microsoft Excel 或 WPS 表格 COM 接口。用户明确批准已检查的预览后，运行：

```powershell
.\.venv\Scripts\python.exe navctl.py commit --yes-reviewed-preview
```

正式写入必须先创建备份，再通过 COM 将方案应用到同目录临时副本，验证临时结果，关闭表格进程，最后原子替换正式文件。任何失败都必须保证正式文件哈希不变。

零新增基线批准不是正式写入：`commit --yes-reviewed-preview` 返回 `changed: false` 和 `approved_baseline: true`，只记录自动更新批准，不创建备份或改变正式表。不得把“无新数据”解释成“无需首次批准”，也不得另写一个有数据就直接落表的脚本绕过批准流程。

这次成功写入会自动批准当前写表配置用于后续自动更新。新增/恢复路由以及列、口径、校验规则或工作簿路径发生变化时，批准自动失效，AI 重新生成一次预览即可；通过 `products pause` 缩小范围且仍有其他活动产品时，程序自动延续原批准。

提交前提醒用户关闭正在打开正式工作簿的 Excel/WPS 窗口。若文件仍被占用，程序必须返回中文结构化错误、删除本次失败备份并保留正式文件；不得强制关闭用户的表格进程。

不得用仅依赖 openpyxl 的方式回写正式工作簿。COM 不可用时，应停在预览阶段，并说明正式写入尚未验证。

不得把 `doctor` 的 `spreadsheet-com: false` 直接解释成“没有安装 Excel/WPS”。先读取 `runtime_platform`、`spreadsheet_apps_detected` 和 `spreadsheet-com.detail`：

- 若不是 `win32`，说明 AI 当前在 WSL、容器或非 Windows Python 中运行；改用 Windows PowerShell 和原生 Windows Python 重新部署。
- 若已检测到 Excel/WPS 但 COM 启动失败，由 AI 主动打开该软件一次，让用户只处理可能出现的首次启动、许可或登录界面，关闭后重跑 `doctor` 和虚构 `demo commit`。
- 若未检测到，也只能报告“尚未确认受支持的 COM 注册”，继续检查软件实际路径和 COM 注册；不得仅凭一次失败断言软件未安装。

只有虚构 `demo commit` 真实通过并报告 `Excel.Application` 或 `ket.Application`，才能确认该电脑具备正式写入能力；AI 在最终交付前还必须重跑 `doctor`，确认授权码、路由、COM 和定时任务均达到相应就绪状态。

## 安装后续自动更新

第一次写入或零新增基线批准成功后，AI 使用首次配置时已经由用户确认的 `schedule` 安装 Windows 自动更新任务：

```powershell
.\.venv\Scripts\python.exe navctl.py schedule install
```

任务使用计划任务程序的交互模式，因此要求用户保持登录。每次运行自动读取邮件、验证、备份并写入正式工作簿；内部临时验收副本在运行结束后删除，不生成需要用户打开的预览。无新数据时不写文件，冲突或异常时停止并记录错误，原表保持不变。

任务名包含运行实例 ID。重复安装只会替换本运行实例记录的任务。删除任务：

```powershell
.\.venv\Scripts\python.exe navctl.py schedule remove
```

使用 `navctl.py schedule status` 查看任务、上次/下次运行、最近一次写入结果和备份路径。普通用户也可双击根目录的 `查看状态.bat`；计划任务和 `手动更新.bat` 共用更新入口，写入前或写入中的失败详情都会保存在根目录 `logs/update-YYYYMMDD.log`，AI 排查即可。

状态输出必须把 Windows 计划任务时间明确标为目标电脑本地墙钟时间，并同时显示系统时区；不得把 COM 返回的本地计划时间误标成 UTC。解析失败诊断只记录路由、解析阶段、附件数量/类型、稳定短指纹和受控异常类型等脱敏元数据；任意异常正文也可能含路径、产品或净值，因此不得写入诊断，更不得记录完整主题、附件名、正文或授权码。

不要为 UNC/网络路径中的运行目录安装任务，也不要复制其他电脑的 Python 路径或任务定义。

删除运行目录前，先移除任务和 DPAPI 密钥：

```powershell
.\.venv\Scripts\python.exe navctl.py schedule remove
.\.venv\Scripts\python.exe navctl.py secret remove
```

备份和日志保留数量由 `app/config.json` 限制。

## 升级已部署环境

不得直接覆盖旧运行目录。先移除旧计划任务，再用最新版 `scripts/bootstrap.py` 在新的短路径创建运行目录；以新生成的 `app/config.json` 为基础迁移 AI 已复核的配置字段，把本地解析器逐个复核后迁入 `app/parsers/`，并保留新的 `runtime_id`。重新执行 `secret launch` 并用 `secret status` 复查，再完成离线演练、验证和一次预览，成功后安装新任务并停用旧目录。不要复制 `app/.venv`、密钥、`plan.json`、`discovery-snapshot.json`、`automation-approval.json`、`run.lock`、预览、日志、备份或状态文件。
