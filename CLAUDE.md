给在本仓库工作的 AI agent / 贡献者的工作指引。先读 [README.md](README.md) 看“是什么”，读 [METHODOLOGY.md](METHODOLOGY.md) 看“为什么这么设计”。本文件讲**怎么安全地改它、以及前人踩过哪些坑别再踩**。

> 本仓库是从真实私募净值项目脱敏抽离的参考实现。所有基金名/代码/邮箱都是占位符（`基金01`、`DEMO0x`、`*.example.com`）。**任何提交前都要复查零真实信息残留**（真实授权码、真实基金名、真实邮箱、真实路径一律不许进库）。

## 1. 架构地图

流水线（每步独立脚本，可单跑单调试）：

```
build_index → fill_index → validate → write → notify
  抓邮件索引    拉指数(可选)  回归校验   写入(先备份) 汇总邮件
```

| 文件 | 职责 | 谁拥有“真相” |
|---|---|---|
| `navlib.py` | 公共库：IMAP 连接、正文解析、主表路径解析、registry/config 读取 | — |
| `phase2.py` | 附件类（.xls/.xlsx/PDF）解析器 | — |
| `build_index.py` | 扫邮箱 → 生成 `index.json`（每封邮件：来源/代码/日期/sheet） | — |
| `propose.py` | **提案引擎**：算出每张表“该补哪些行”，只读不写 | — |
| `write.py` | 唯一**写净值**的地方；写前自动备份 | ✅ 净值数据 |
| `build_registry.py` | 从 Excel 表**重新生成** `registry.json` | — |
| `validate.py` | 回归校验：用解析器复现已记录的净值，证明取数正确 | — |

**单一真相源 = 那份 Excel 主表。** `registry.json` 是从主表**自动生成**的派生物，邮件↔表的匹配靠 **A 列产品代码**（不是中文表名）。记住这条，下面的运维设计才讲得通。

## 2. 运维设计（无 AI、非技术运营人员也能做）

这是本项目最重要的实战经验：把“天天要做的”和“偶尔要做的”彻底分开，前者零编辑，后者只改一个地方。

| 要做的事 | 操作 | 碰不碰代码/配置 |
|---|---|---|
| **加数据**（每周新净值） | 定时任务自动跑；或双击 `run_weekly.bat` | ❌ 完全不用 |
| **改名字**（改表显示名、加后缀） | 在 Excel 里改**表标签**（A 列代码别动）→ 重跑 `build_registry.py` | ❌ 不手改 registry |
| **加新产品** | Excel 复制一张表→改标签→A 列填代码、B 列填名称→重跑 `build_registry.py` | ❌ 不手改 registry |
| **换邮箱授权码** | 只改 `config.json` 的 `password` 一行 | ✅ 仅这一行 |

**铁律：永远不要手改 `registry.json`。** 它是 `build_registry.py` 从主表生成的；手改 = 表名和配置两处不同步 = 崩。改主表 + 重建，是唯一正确路径。

三道护栏（改任何东西都靠它们兜底）：
1. **先预览后提交**：`write.py`（无 `--commit`）只打印将写入的行；确认后才 `--commit`。
2. **写前自动备份**：`write.py --commit` 先存带时间戳副本到 `backups/`，出错能回滚。
3. **异常不自动写**：解析失败/周收益异常/疑似赎回 → 只标记，等人工确认。

交接给同事时，把上表 + “求助清单”（报错、新托管商、回滚 → 找开发者）写成一页纸贴墙即可。

## 3. 发布 / 迁移（开发机 ≠ 运行机，分清代码和数据）

实战里代码在开发机上改、跑在公司电脑/公盘上，而且**代码和数据都改得很勤**（修解析器、加产品、改配色…）。最大的坑不是“怎么把代码送过去”，而是**整包覆盖会把运行环境里的真相数据盖掉**。

**单一真相是运行机上的那份 Excel 主表**（操作者的手改格式、累计行、定时任务每周自动写的净值，都在它身上）。开发机包里的主表只是副本。由此一条铁律：

> **代码同步绝不携带数据文件。** 发布只覆盖程序（`*.py`），永远排除：主表 Excel、`config.json`（含授权码）、`registry.json` 等运行时状态。

落地方式（够用就好，别上工程）：

| 改动类型 | 实际动什么 | 怎么同步 |
|---|---|---|
| **改代码**（修 bug / 逻辑） | `*.py` | 只覆盖脚本；显式排除主表 / `config.json` / `registry.json` |
| **加 / 改产品**（动数据） | 主表 Excel + 偶尔 `config.json` | 主表在运行机端是真相，**不从开发机覆盖回去**；配置量小，明确告诉操作者“改哪一行/拷哪个文件”，手动放 |

推荐做一个 `更新代码`脚本：只 `robocopy` 程序文件（带排除清单）+ 覆盖前把旧代码目录压一个带时间戳的备份（可回滚）。首次安装/换机用另一个一键脚本（装依赖、建定时任务）。两个入口分开，权责清楚。

**不要过度设计。** 这类项目（真相是 Excel、单人运营、可能连 PowerShell 都不让跑）**不需要** SQLite + 数据库迁移、`releases/` 多版本包、`deploy.ps1` 七步切换、`config_version` 校验——那是给多人频繁发版、要审计回滚的团队用的，搬过来纯属负担。频繁修改真正需要的只有一件事：**代码与数据互不覆盖**。一个排除清单 + 一次备份就解决。

> Windows 批处理 / 公盘(UNC) 部署另有一组硬坑（`.bat` 行尾必须 CRLF、内容纯 ASCII、UNC 靠 `pushd` 自动映射盘符、需管理员用“右键运行”闸门而非脚本自动提权），见 METHODOLOGY 的相关说明 / 第 4 节踩坑。

## 4. 硬规则 / 环境坑（默认方案常常是错的）

1. **openpyxl 写 Excel 会毁掉图表/图片/透视表/VBA**（实测：某含图表的表 9 图→剩 3 图）。
   - **含图表的工作簿，绝不能用 openpyxl 写**——必须用 **Excel COM**（`win32com.client` 或 `xlwings`）驱动真实 Excel 逐格写、保存。
   - openpyxl 只适用于**纯数据、无图表**的表。写之前先查 `ws._charts` / `ws._images`。
2. **各托管商格式各不相同**，解析器要对着真实邮件逐家做，没有通用 parser；本仓库的解析器是脱敏示例，接你自己的来源要重写。
3. **指数/外部取数源依网络而定**，公开 API 不一定通（原环境东方财富被墙→换腾讯）。别假设能连上；要有兜底源 + 用表里已有值自校验。
4. **每只基金净值口径可能不同**、账户明细科目映射、周净值逻辑都按具体产品确认，别用统一口径硬套。
5. **序列化日期 / 时区要统一**（Excel 序列日期自 1899-12-30 起算），写入前归一处理。

## 5. 我（前几任 Claude）犯过的错——别再犯

诚实记录，因为这些都真金白银耽误过事：

- **过度设计安全把能跑的系统搞挂了。** 我擅自把邮箱授权码从 `config.json` 挪到 `secret.json`“更安全”，结果迁移到没有 `secret.json` 的公司电脑上直接登录失败、全线瘫痪。
  → **教训：满足操作者最简需求。** 用户明确要“授权码就放运行文件里，粘贴即用”，就照办，别加没要求的“加固”。改动用户的运行环境前，先确认在目标机器上仍能跑。
- **在看不见的机器上盲调。** 用户的报错我看不到现场，靠猜来回改，慢。
  → 让用户贴真实报错/截图，或先在能复现的副本上调，别盲改生产。
- **一次性迁移脚本里写死了一个净值数字。** 我在“清排版”的补丁里硬编码了某基金净值（还是错的），差点污染真表。
  → **`write.py` 才拥有净值数据；一次性迁移/排版脚本绝不许写净值。** 各司其职。
- **“没改几行代码”不等于“过程很快”。** 最后改动很小，但定位花了很久——因为隔着看不见的机器、隔着乱码控制台。
  → 复杂的是诊断不是补丁；先把可见性建好（日志、预览、能复现）。

通用反模式见 [METHODOLOGY.md](METHODOLOGY.md) 的“反模式”一节。

## 6. 约定

- **中文金融配色**：涨=红、跌=绿（与西方相反），别写反。
- **GBK 控制台乱码**：Windows 控制台打印中文可能花屏，那只是显示问题，**数据本身没事**；脚本里已设 `PYTHONIOENCODING=utf-8`。别为这个去“修”数据。
- **格式跟着文档走**：新行复制上一行的样式，不在代码里写死字体/颜色——操作者要改样式只在 Excel 里改。
