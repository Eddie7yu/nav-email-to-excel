# 运行目录分层规范

这项规范面向最终使用自动化的普通用户。目标不是让用户理解代码，而是让他打开运行目录后立刻知道“点哪里、去哪里看结果”，同时让 AI 仍能维护完整程序。

## 固定结构

`scripts/bootstrap.py` 默认在目标工作簿旁生成独立运行目录，并保持以下结构：

```text
D:\data\
├─ nav.xlsx
└─ 净值自动化\
   ├─ 使用说明.txt             # 用户先读
   ├─ 首次授权.bat             # 第一次部署时由 AI 主动弹出
   ├─ 查看状态.bat             # 查看计划任务和最近结果
   ├─ 手动更新.bat             # 必要时手工触发一次安全更新
   ├─ .gitignore               # 防止本地配置、日志和工作簿被误提交
   ├─ previews\                # 只放第一次验收或规则变化后的预览
   ├─ backups\                 # 正式写入前的自动备份
   ├─ logs\                    # 自动任务日志
   └─ app\
      ├─ navctl.py
      ├─ config.json
      ├─ assets\nav-standard-cn.xlsx   # 内置脱敏模板，只由程序读取
      ├─ parsers\
      ├─ .venv\
      └─ 其他内部源码、状态与定时脚本
```

正式工作簿仍保留在用户指定的原路径，不复制到运行目录根部。没有现成工作簿时，程序也只在用户指定的新 `.xlsx` 路径创建；`app/assets/` 中的是无真实数据的只读模板资产，不是正式表。一个运行目录只服务配置中登记的那一份工作簿；默认的 `净值自动化/` 已存在时，引导程序拒绝复用或覆盖。

## 普通用户边界

普通用户只需要知道四件事：

1. 第一次授权时，在 AI 主动弹出的窗口粘贴授权码；
2. 第一次配置时打开 `previews/` 中名称带“只读审查”的预览验收；它应以只读方式打开，窗口标题不能是正式工作簿；
3. 日后双击 `查看状态.bat` 查看授权、路由、基准待审、批准、任务和最近结果，必要时双击 `手动更新.bat`；
4. 需要回退或排错时由 AI 查看 `backups/` 和 `logs/`。

不得要求普通用户进入 `app/`、修改 JSON、运行 Python、挑选解析器或理解虚拟环境。

如果预览生成后正式工作簿被用户、Excel/WPS、同步软件或其他进程保存，旧预览自动失效。AI 读取本地 `app/concurrency-report.json` 中不含单元格值的 Sheet/行列/坐标差异，核实当前文件后重新生成预览；不得回滚或覆盖外部保存，也不让用户自己合并两个文件。

## BAT 闪退与命令无法识别

典型错误反例：首次部署后，用户双击中文 BAT，窗口立即消失；或者窗口显示某个命令“不是内部或外部命令，也不是可运行的程序或批处理文件”。Agent 因看不到完整错误，便声称电脑不支持 BAT、要求用户反复双击，或临时新写一个直接调用系统 `python` 的脚本。

这种处理是错误的。常见根因包括：系统 Python 没有加入 PATH、BAT 与 `app/.venv` 来自不同版本、入口无法进入运行目录、控制器文件缺失，以及运行路径中的空格、中文、`&` 或括号被 `cmd.exe` 错误拆分。正确处理：

1. AI 在原生 Windows 终端解析 `py.exe -3` 或 `python.exe` 的真实可执行命令，不让用户输入部署命令；
2. 只接受引导程序一次生成的完整运行目录，不把新版 BAT 覆盖到旧版 `app/`，也不反向复制；
3. 用户入口只调用本目录的内部入口；`手动更新.bat` 与计划任务共同调用 `app/run-update.cmd`，缺失时显示明确原因并暂停留窗，写入前失败也必须留下 `logs/update-YYYYMMDD.log`；
4. 授权入口由程序以 BAT 文件名配合其工作目录启动，不把包含命令行特殊字符的完整路径拼进 `cmd /c`；
5. 在完全虚构的临时运行目录中通过真实 `cmd.exe` 执行三个入口，检查退出码和输出后，才声称入口可用。

不得通过删除 `pause`、使用系统 PATH 中未核实的 `python`、移动 `.venv`、关闭错误输出或让定时任务直接写正式表来掩盖问题。旧运行目录出现混合版本时，按升级流程新建完整运行目录并重新验收。

## AI 和开发边界

- 所有 `navctl.py` 命令都从 `app/` 执行。
- 配置和内部状态只写入 `app/`；本地解析器只写入 `app/parsers/`。
- 探索阶段允许 Agent 自主编写临时 Python/PowerShell、比较解析库、保存中间诊断和对工作簿副本试验；这些材料只能放在 `app/diagnostics/`，不得直接写正式表或安装为任务，问题解决后、交付前删除。
- 不得在根目录新增 `.py`、`.ps1`、`.json`、抓包文件、邮件导出、测试工作簿、`debug-*` 或 `test-*` 文件。
- 不得把 Skill 仓库、开发测试、真实工作簿或附件存档复制到运行目录根部。
- 不得移动或重命名四个中文入口、三个结果目录或 `app/`；计划任务和升级流程依赖固定结构。

## 交付前检查

AI 在告诉用户“部署完成”前必须确认：

- 根目录没有散落的 Python、PowerShell、JSON 或临时文件；
- 四个中文文件、根目录 `.gitignore` 均存在，三个结果目录和 `app/` 均存在；
- 三个 BAT 均为 ASCII、CRLF，并在真实 `cmd.exe` 中通过中文、空格和命令行特殊字符路径测试；`首次授权.bat` 能弹出真实终端，`查看状态.bat` 能显示状态，所有错误分支都会说明原因并留窗；
- `查看状态.bat` 至少显示授权是否可用、配置/活动/暂停路由数、基准来源与许可阻塞数、首次批准状态、任务数以及最近更新时间和结果；只显示任务是否存在不算交付完成；
- `手动更新.bat` 只调用 `app/run-update.cmd`，不得直接调用 `navctl.py scheduled-update`；批准前拒绝、配置错误和正式更新使用同一脱敏日志入口；
- `app/.venv`、`app/config.json` 和 `app/parsers/` 位于内部层；
- `doctor` 与虚构 `demo commit` 通过，第一次真实预览路径明确交给用户；
- 用户无需知道内部目录结构，也能完成授权、验收和日常查看。

升级时新建另一套短路径运行目录，复核后迁移必要配置和本地解析器；不要把新版文件覆盖进旧 `app/`，也不要把旧目录中的临时文件带入新环境。
