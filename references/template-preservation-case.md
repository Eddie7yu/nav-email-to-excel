# 反例：不要把现有净值模板改造成另一张表

部署已有工作簿、邮箱历史比表内历史更完整、邮件频率与表格频率不同，或验证失败时，必须先读本文件。

## 脱敏案例

目标工作簿有两个产品页：

- 产品 A：表头下只有“产品代码 + 预留日期”，净值为空，下一行是“累计”。
- 产品 B：已有十余条周度历史，带周收益、指数、指数收益、超额和“累计”行。
- 邮箱中产品 A、B 都有充足历史邮件，但邮件按日发送。

失败的 Agent 为了让验证通过，清空两张表的原数据，把 `summary` 改成 `append`，再把所有日度邮件写入；最后“累计”行、周收益公式、指数和超额全部消失。它生成了文件，但没有完成用户要的工作。

## 唯一正确的理解

邮件决定“有哪些真实产品、日期和净值”；现有 Excel 决定“这些数据写到哪里、多久一行、哪些公式和汇总必须保留”。

只要邮箱中能唯一识别该产品的历史邮件，就主动补录，不要求用户手工增加历史净值，也不因为表内历史为空而放弃。补录时必须服从模板：

- 原表是周度，就仍按周度落表；日度邮件可全部用于核验，但不能未经允许全部变成日度行。
- 原表是日度，就按日度落表。
- 原表有稳定历史日期规律时，沿用该规律；只有表头明确“周度”但没有足够历史可推断日期时，默认选每个已完成自然周最后一条可用净值。
- `return_frequency` 只控制收益公式周期，`data_frequency` 才控制实际写入日度还是周度行；两者不得混为一个开关。默认保持 `data_frequency: auto`，让运行时以模板历史和表头判断。显式设置若与模板冲突，必须停止。
- 计划任务多久运行一次，与工作表多久保留一行是两件事，不得混淆。
- 预留行没有净值时，其日期不是权威历史；用邮箱中最早一条真实记录替换该预留行，再在汇总行前继续补录。

## 禁止操作

- 第一次预览获批前，不得修改、清空、另存覆盖或重排正式工作簿。
- 不得因为 `summary` 验证失败就改成 `append`。
- 不得删除既有历史、公式、累计/合计行、基准列、超额列或工作表来制造“验证通过”。
- 不得把周度模板改成日度模板，或反过来。
- 不得放弃模板已有列并把“只有净值写进去了”称为完成。
- 不得用 openpyxl 或临时脚本直接改正式表；只能生成独立预览，获批后走 COM 提交流程。
- COM、公式、基准或定时任务尚未完成时，不得声称已经部署完成。

## Agent 应当怎么做

1. 只读检查原工作簿，记录工作表、表头、日期规律、既有历史、公式列、汇总行、基准列和样式。
2. 从邮箱发现同一产品的全部历史邮件，按代码/名称和主题精确分流。
3. 以模板的数据频率筛选拟写入记录；不改变原来的周度/日度口径。
4. 对只有产品标识、日期但净值为空的预留行，用最早真实邮件补全；保留其后的“累计/合计”行。
5. 在独立预览中补历史，并延续相邻行公式、格式和基准逻辑。
6. 对比原表与预览：原有单元格、公式、汇总行、基准列和工作表不得减少；只允许出现经核实的新增历史记录和由此延伸的受管公式。
7. 让用户只检查第一次预览；批准前正式表必须保持原文件哈希不变。

如果现有运行时代码无法在保留模板的前提下完成，正确结果是停止并修复运行时，而不是简化用户的工作簿。

## 交付前快速检查

- 邮箱中该产品的可用历史是否已经按模板频率进入预览？
- 原表已有日期和净值是否仍在，且顺序未被擅自改变？
- 原公式数量和公式列是否保留并正确延伸？
- “累计/合计”行是否仍在正确位置？
- 指数、指数收益和超额列是否按原模板保留？
- 预览是否仍然一眼看得出是用户给的那张表，而不是 Agent 另做的一张表？

任一项为“否”，预览不得交给用户验收，更不得安装自动更新任务。
