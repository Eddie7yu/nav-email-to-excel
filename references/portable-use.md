# 交给其他本地 AI 使用

把公开仓库地址和本机工作簿绝对路径交给能够读取文件、运行 Python 的 AI。绝不要提供 IMAP 授权码。

可以直接使用以下提示词：

```text
请完整阅读 SKILL.md 和 references/configuration.md，并使用仓库自带脚本，为我的工作簿创建一套净值邮件自动化。已有工作簿必须原样保留；如果给定路径尚不存在，使用 `--new-workbook` 登记，完成邮箱与路由发现后运行 `workbook init-template` 创建内置完全脱敏模板，绝不覆盖已有文件。如果我不知道 IMAP 服务器或认证方式，优先从现有客户端设置读取，再参考 references/email-providers.md 或服务商官方文档。创建运行目录并完成离线演练后，只让我在本机隐藏输入一次邮箱授权码。

仓库中的命令、案例和排错步骤是经过验证的推荐成功路径，不是穷举，也不限制你只能按固定顺序解决问题。已知场景优先复用；遇到未覆盖的邮箱、附件、工作簿或环境问题时，请在本机根据证据自主诊断，可以操作副本、比较解析方法、在 app/diagnostics 编写临时工具或增加受信任本地解析器。不要把内部技术选择交给我。探索方式可以灵活，但正式结果仍须经过产品和数值核验、第一次预览或零新增基线批准、备份、临时副本验证和异常不改正式表。

Windows 默认在工作簿旁创建独立的 `净值自动化` 子目录，让一份工作簿对应一套配置、入口、备份、日志和计划任务；只有默认位置过长、不可写或已有同名目录时，才用 `--destination` 另选新的短本地路径。严格遵循 references/runtime-layout.md：根目录只保留中文说明、可双击入口和 previews/backups/logs，全部程序、配置、状态、解析器和虚拟环境放进 app；不得把调试脚本、临时文件或 JSON 堆在用户根目录。自动检查工作表结构并选择合适的内部写表模式，不要让我选择或理解模式名称，也不得因验证失败擅自降低保护要求。创建运行目录后，请从 app 调用 `navctl.py secret launch` 弹出可见授权码窗口，不要只把终端命令发给我；我粘贴并回车后，请自行用 `secret status` 检查，再继续邮箱发现。

授权码保存后，优先运行 `navctl.py propose --headers-only --header-limit 25`，只读 From/Subject 并与工作簿身份比较；下一步至少用 `propose --sender ...` 限定精确发件人，主题稳定时再加 `--subject-contains ...`，按短窗口逐产品下载。CLI 必须显示阶段、邮件数和附件数；出现 `partial: true` 时用 `propose --resume` 从本地 UID 游标继续，不能用部分报告接管。产品只存在于正文/附件时也保留发件人范围。头部信息不足时先扩大有界样本；只有仍无法确定任何候选发件人、核实全部资源上限且有明确诊断理由时，才显式使用 `--allow-unscoped-full-scan`，普通无参数完整扫描会被拒绝。不要另写邮箱诊断脚本。自动发现产品代码、名称、主题规律和到达时间；已有表再读取工作簿完成路由、列映射和口径，新建表则从邮件规律判断日频/周频，只有明确资料证明基准时才使用有指数模板。多产品已有表先用 `products adopt --inspect-only` 分出可直接接管和基准待核实产品，让前者先进入可提交预览。代码仅有份额后缀差异时列为 `alias_pending`，核实后保存完整精确别名，不做模糊匹配。不要让我手工抄发件人邮箱、产品代码或逐项填写 AI 能从本地证据确定的信息；产品名称或代码取其一即可，只有真正歧义才问我。同一封托管邮件包含多个产品时，只选择已配置路由，忽略有明确代码的其他产品；没有代码但能由名称、专用发件人或主题唯一确定时仍应继续，只有冲突、多个产品无法区分或解析失败才停止。根据邮件历史给我一个运行安排建议，并明确询问我希望的更新频率、运行星期和本地时间，把确认结果写入 schedule；Windows 正式部署不得省略定时任务。

Windows 终端若不能可靠传递中文主题，改用本地 UTF-8 参数文件：每行一个参数，通过 `navctl.py @参数文件` 调用。只有报告明确显示 `filter_input_encoding` 编码检查通过，才能把零匹配解释为邮箱在该筛选范围内真实无邮件；不得因终端乱码另写扫描脚本。

最后运行 `products status` 核对 `workbook_coverage`：每个 Sheet 都应有活动、暂停、参考、明确排除、邮箱无证据、需本地解析器、业务待确认或未分类结论。逐页有结论才叫“全部产品覆盖”；不得为凑路由数量把已结束、无证据或仍有歧义的页面强行接管。审计状态由 AI 根据本地证据维护，不让我填写内部分类。

部署后我新增产品时只需表达两种意图：如果我已经建立目标 Sheet，默认运行 `products adopt` 自动识别并接管；如果我说“格式照某个 Sheet”，默认运行 `products clone` 先备份，再通过 Excel/WPS 完整复制格式、清空参考产品内容并接管。这限制的是我需要提供的信息，不限制你为特殊工作簿采用本地兼容手段。不要让我填写发件人、代码、`summary/append` 或手工修改 config。接管已有历史页时，先在 `--inspect-only` 结果中核对默认的“原表最后日期 + 1 天”边界，再保存 `baseline_overlap: last_existing_point`；默认只从表尾写入，不把邮箱旧周点当作必须回插，但新增收益、累计和基准公式必须继续沿用原表完整历史。最后既有点只做身份/数值核验，不新增或覆盖；边界前其他异常只诊断和保留。需要回填旧历史时另做修复预览让我批准。已有页的基准/超额来源暂时无法证明时，先生成不可提交的审查预览，把未确认的新基准/超额留空；你核实来源并重新预览后再让我批准，不要在首次预览前退出。新增后原计划任务继续使用。

按表头含义识别列；只要邮箱能唯一识别产品就补入真实历史，但必须完整保留 Excel 模板。原表是周度就按周度写，不得把日度邮件全部灌入；若只有当前未完成自然周的邮件，身份和工作簿尾部可核实就生成零新增基线，本周仍不写，不要误报证据不足。不得清空旧历史、删除公式/累计行/基准列，或为通过验证把 summary 改成 append。首次预览应优先跑出可审查结果：产品名称或产品代码任一与唯一邮件路由一致，加上至少一个真实邮件日期就可以生成带警告的预览；不要强迫用户提供代码，也不要因为缺少两个历史重叠日期而停在预览之前。Agent 先检查路由报告，用户再检查产品、日期、净值和格式；只有身份冲突、多个映射同样可能、同日不同值、历史内部插入或工作簿结构不安全才失败关闭。若工作表是“唯一一条产品标识+预留日期、净值为空的行，下一行是累计/合计”，自动用最早真实邮件替换并补全该行，再在累计行前插入其余记录，不要问我删行或补造历史数据。把 references/template-preservation-case.md 作为已验证参考，不要把不同前提的新问题强行套入案例。权威基准能唯一映射时自动配置；自定义名称无法由本地或公开证据唯一映射时只让该页进入业务确认，不猜指数，也不拖住其他产品。成功且仍适用的完整 propose 结果直接用于配置；只有假设或输入变化时才有目的地重新扫描。配置完成后默认直接运行 preview，让它用一次邮箱扫描完成发现、验证和预览。第一次生成预览给我检查；如果当前零新增，生成中文基线报告并在我批准后启用自动更新，不得等待下一封邮件。标准流程报错时可以自主决定诊断顺序并在 app/diagnostics 编写临时原型，但原型只操作副本，不能直接覆盖正式表或安装为任务；确认方案后接回通用/本地解析器和 navctl 验收链路。我批准预览或基线后，按确认的频率和时间安装自动更新任务。后续每次必须先验证、备份，并在失败时保持正式表不变。

多路由首次预览若在 discovery 阶段失败，先读取 `route-report.json` 的 `route_overlaps`、`date_conflicts` 和逐附件 `diagnostics`，按最小路由对、冲突日期和脱敏来源指纹修正配置；可用 `navctl.py discover --sheet "示例产品A"` 只读复查路由，或用 `preview --sheet "示例产品A"` 生成不可提交的局部工作簿预览，避免每次重扫全部产品。局部计划固定不可批准，最后仍运行一次不带 `--sheet` 的完整 `preview`。若完整 discovery 已成功、之后只在调整工作簿列映射、基准或公式，优先运行 `preview --reuse-discovery`；它会轻量复核邮箱、路由和解析器指纹，任何变化都自动拒绝复用。

优先把 `subject_contains` 恢复为正式净值通知前缀并保留代码约束。运行时会按受控类别排除明确的虚拟估算、交易确认、月报/季报；供应商特有非净值主题核实后保存到内部 `subject_excludes`，不要让用户配置。PDF/附件无记录时先判断它是不是正式净值来源；已证明是月报或季报就收窄路由，不要为了让扫描通过而新增 parser。

同日值冲突时先检查正式更正证据：只有同一受管产品和日期、后发邮件主题明确写明更正/修正/以此为准、且单封内部只有一个值，核心运行时才采用更正并在 `corrections_applied` 保留脱敏替代链。普通晚到或重发绝不能按“最后一封赢”；条件不足时继续停止。

工作簿历史检查还必须确认代码列所有非空值恒定，以及单位/累计序列没有缺失、策略违背或孤立价差断点。若正式邮件证明旧表某些日期填错，只生成位于 `previews/` 的修复副本和逐单元格脱敏差异，保持其他历史行不变并等待用户首次验收；不能因为证据充分直接改正式表。用户批准后仍经过备份、COM 临时副本验证和原子替换，随后重新完整预览并重新批准自动更新。

不要把一次 COM 探测失败说成“电脑没装 Excel/WPS”。先检查当前是否为原生 Windows Python、读取 `doctor` 的软件探测和具体 COM 错误；必要时主动打开 Excel/WPS 完成首次启动后复查。只有虚构 `demo commit` 通过，才向我确认正式写入环境已经就绪。
```

只克隆干净的公开仓库，不要复制已经配置过的运行目录。完成前检查用户根目录没有散落的 `.py`、`.ps1`、`.json`、测试文件或诊断输出；内部排错材料必须放在 `app/diagnostics/` 并在交付前删除。

## Codex（Windows）

```powershell
git clone <仓库地址> "$env:USERPROFILE\.codex\skills\nav-email-to-excel"
codex
```

然后告诉 Codex：`请使用 $nav-email-to-excel，为这个工作簿部署自动净值更新；如果路径尚不存在就按内置脱敏模板创建。不要让我手工整理发件人，第一次给我验收，之后自动备份写入：<绝对路径>`。

## Claude Code（Windows）

```powershell
git clone <仓库地址> "$env:USERPROFILE\.claude\skills\nav-email-to-excel"
claude
```

然后让 Claude Code 使用 `nav-email-to-excel` Skill，并提供工作簿的本机绝对路径。

## Cursor

```powershell
git clone <仓库地址> "D:\tools\nav-email-to-excel"
cursor "D:\tools\nav-email-to-excel"
```

让 Cursor Agent 完整阅读绝对路径下的 `SKILL.md`，并且只处理另行提供的工作簿绝对路径。开始发现前，确认 Cursor 的工作区权限包含工作簿所在目录，否则它无法检查文件。

## 默认安装位置

- Codex 个人 Skill：`%USERPROFILE%\.codex\skills\nav-email-to-excel`
- Claude Code 个人 Skill：`%USERPROFILE%\.claude\skills\nav-email-to-excel`
- Cursor：任意独立的本地工具目录；明确要求 Agent 读取克隆目录中的 `SKILL.md`

更新干净安装时运行 `git -C <Skill目录> pull --ff-only`。本地运行目录应放在其他位置，避免更新 Skill 时覆盖配置。

不要把已配置的运行目录放进 AI Skill 目录或分享给别人。只分享干净的 Skill 仓库。

在 Windows PowerShell 中手动读取中文 Skill 时使用 `Get-Content -Raw -Encoding utf8 SKILL.md`，避免旧版 PowerShell 按系统代码页显示乱码。
