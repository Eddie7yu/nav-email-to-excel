# 把「邮件里的数据」安全写进「活的电子表格」——一套通用方法论

> 源自一个真实项目：从托管邮件中提取私募净值，自动写入运营人员手工维护的 Excel。
> 本文抽掉所有业务细节，提炼成可复用的设计与工程方法论。适用于任何
> 「半结构化输入（邮件 / PDF / 附件）→ 人工维护的系统记录（Excel / Sheet / DB）」的自动化。

---

## 0. 核心命题

你要自动化的对象，往往是一份**活的、人在用、出错代价高**的文件。
难点从来不是"读出数字"，而是：

- 输入是**半结构化**的（不同来源格式各异、还会变）；
- 目标是**人工维护**的（有手填怪癖、随时被改、不能写坏）；
- 一旦写错，**后果真实**（财务 / 合规 / 信任）。

所以整套方法论围绕一个词展开：**可信地、可回退地、不打扰人地写入**。

---

## 1. 八条设计原则

### ① 读优先、写最后、提交前必预览
每个会碰到"事实源"的环节都分两挡：**预览**（写到副本，原文件不动）和 **`--commit`**（正式写，且**先自动备份**）。
默认永远是预览。让操作者眼睛确认过，再提交。

### ② 把流程切成独立阶段，各自可单独跑
`抓取 → 解析 → 校验 → 写入 → 通知`。
每一步是独立脚本、独立可运行、独立可调试。出问题能精确定位到某一阶段，而不是面对一个黑箱。

### ③ 校验是一等公民：先证明"解析对"，再相信"解析新的"
写新数据前，先拿**最近 N 行已知正确的数据**，让解析器重新解析一遍，确认能**复现**人工已记录的值。
复现得了 → 解析逻辑可信 → 才允许它处理新数据。这一步能挡住绝大多数静默错误。

### ④ 永远不要信任手工维护的数据
人会留下：单元格尾部空格、隐藏的换行符 `\n`、预填的空行、参差的字体。
对策：
- **解析端防御**：把"只有空白"当作空；容忍多余字符。
- **写入端净化**：写之前 `strip()`，别把脏字符传播下去。
- 把每一个踩过的坑写成注释，因为它们**不可预测，只能撞见**。

### ⑤ 格式跟着文档走，不跟代码走
新行的样式**复制上一行**（字体、颜色、边框、数字格式全抄），**不要在代码里写死**。
这样操作者改格式只需在表里改最后几行，下次自动延续——**呈现层归人，数据层归代码**。

### ⑥ 幂等 + 容错：一个坏表不能拖垮整轮
- 目标表名对不上 / 暂缺 → **跳过并记录**，不要抛异常崩掉整轮。
- 任何脚本重复跑都应安全（幂等）。
- 网络等外部依赖失败 → 该项**延后重试**，而不是写入坏数据。

### ⑦ 配置驱动，而非改代码
表名、识别码、来源映射、处理范围，全部放 JSON 配置。
**加一个新数据源 = 改配置，不是改代码。**

### ⑧ 一次性迁移 与 周期任务 严格分开
改名、重排版、补列这类**结构变更**，做成"跑一次即删"的独立脚本；
**绝不放进定时任务**。这样每天自动跑的那条流水线永远只做数据、不会某天突然改坏你的格式。

## 2. 工程落地清单

| 关注点 | 做法 |
|---|---|
| **可移植** | 自动定位目标文件（按特征名/唯一匹配），路径相对脚本，换机器即用 |
| **可回退** | 每次写入前自动备份带时间戳的副本 |
| **可观测** | 每次运行落日志；每条写入进 changelog；跑完发一封汇总 |
| **可自检** | 提供一条命令：检查依赖、配置、连接、目标文件是否就位 |
| **路由有护栏** | 主用唯一识别码匹配；名称兜底；对"歧义匹配"显式拦截 |
| **密钥最简** | 用**匹配操作者真实风险模型**的最简方案，不要过度加固（见下） |

---

## 3. 过程方法论（怎么一步步搭）

1. **先盘点，不写入**：先把所有输入源、目标结构、命名规律摸清，全程只读。
2. **先证明，再扩张**：在已知数据上跑通校验，再让它碰新数据。
3. **最小可部署改动**：预览 → 核对 → 提交，单步前进。
4. **不做没被要求的事**：每多一层抽象/加固都是未来的债。
5. **卡住就问，别猜着实现**：一个澄清问题，省掉一整轮返工。

---

## 4. 反模式（用真金白银换来的教训）

- **过度设计安全 = 亲手搞坏可用系统。**
  曾把密钥从配置挪到"更安全"的独立文件——操作者机器上没那文件，直接连不上、全线瘫痪。
  **教训：密钥处理要匹配操作者的真实威胁模型，而不是工程师想象中的最佳实践。** 操作者说"放配置里就行、不怕泄露"，那就放配置里。

- **在看不见的机器上调试，极慢。**
  真正出问题的环境你往往摸不到，每次"跑一下→报结果→再改"都是高延迟。
  **对策：把自检/诊断做进工具本身**，让操作者一条命令就能把环境状态打印给你。

- **依赖和顺序不讲清，就会被当成"你没做"。**
  "先迁移结构、再写数据"这种前提，一开始就要写死在文档和提示里。

- **代码改得很少 ≠ 过程很快。**
  我们最后真正改的代码极少——恰恰说明架构是对的。时间几乎都花在"撤销自己的过度设计"和"逐个逼出手填数据的怪癖"上。**少改代码是结果，不是过程。**

---

## 5. 一句话总结

> **把人当文件的主人，把代码当谨慎的助手。**
> 助手做事：读优先、改前先备份、动手前先预览、格式跟着主人走、有疑问就问、能一键就别搞复杂。

## 附：开源前的脱敏清单（若连代码一起发布）

- [ ] 配置里的账号/密钥/授权码 → 全部清空，只留占位说明
- [ ] 真实业务标识（产品代码、机构名、来源邮箱）→ 换成示例值
- [ ] 缓存/状态/日志/备份（`index.json`、`last_run.json`、`logs/`、`backups/`）→ 不入库（加 `.gitignore`）
- [ ] 真实数据文件（`*.xlsx`）→ 换成一个脱敏样例表
- [ ] 加 `LICENSE`、`README`（含一条命令的安装/试跑）、`.gitignore`
- [ ] README 里写清"密钥放哪、怎么填"，并给出从最简到更安全的几种选项，让使用者自己选
